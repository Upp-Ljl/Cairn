## 8. 反思

**做对的地方**：subagent rotation 策略在执行层面非常有效。32 个 commit 对应 30 个有实质内容的 task，每个 task 用 fresh subagent 保证注意力聚焦。从 commit message 的一致性和测试覆盖率来看，没有因为上下文积累导致的偏移或遗漏。补丁 2（schema 一次到位）是另一个明显做对的决定：W1 期间直接用 P2/P3 完整 DDL，意味着 P2/P3 执行时只需要增量代码而不需要触 schema，消除了楔期遗留数据迁移这个高风险点。

**可以更好的地方**：push 阶段遭遇 GCM（Git Credential Manager）凭证弹窗，这在 Windows + Claude Code 的 headless 环境下没有预先处理路径，拖慢了 push 确认节奏。下次可以在开工前确认 git push 不需要交互式弹窗（用 SSH 或 PAT 写入 credential store）。另一个偏慢的环节是 T11（Claude Code 手动接入 + smoke test），这一步无法被 CI 或 acceptance test 完全替代，但也没有被事先计划预留时间，导致变成了"文档说步骤但没有执行记录"的状态，T12 dogfood 整体被推到了用户手动执行。

**一个非显而易见的 lesson**：acceptance test 先行（补丁 1）的价值不仅仅在于测试覆盖，更在于它强迫对"什么叫完成"做出精确定义，然后把这个定义固化成可执行的代码。楔期的 7 个工具，如果没有 `wedge-acceptance.test.ts`，"完成"的边界会在对话中漂移——每个 subagent 只看到当前 task 的局部视角，很容易在某个工具的细节上过度工程化，或者在全局层面少做了某条 §17.1 的契约。acceptance test 充当了跨 subagent 的全局契约锚点，这比任何文档描述都更可靠。
