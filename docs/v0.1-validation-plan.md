## 重要原则申明（v3 起效，与 v2 冲突处以此为准）

1. **商品化指标不在本文档范围**：装机量、用户数、telemetry、anonymous metrics、private release、种子用户邀请等运营/销售语言在本文档中**不出现**。"ship"在本文档语义 = 工程交付完成，不是商业发布。
2. **Pet UI 降级为占位形态**：v0.1 范围内 Pet UI 合并为 **U-PET-PLACEHOLDER** 单个 unit，只要求图标在 + 气泡能弹；不做动画系统、不做"忽略 3 次自动静音"逻辑（US-3 AC 第 3 条）、不验证 idle CPU < 1%（US-3 AC 第 1 条）。后两条显式延期 v0.2。
3. **Daemon 是核心（系统级守护进程）**：Daemon 拆为 11 个子 unit（U-DAEMON-PROC / U-DAEMON-IPC / U-DAEMON-PERSIST / U-DAEMON-WATCHDOG / U-DAEMON-SUBAGENT-SPAWN / U-DAEMON-SUBAGENT-SUPERVISE / U-DAEMON-SUBAGENT-KILL / U-DAEMON-SUBAGENT-IPC-BRIDGE / U-DAEMON-LOG / U-DAEMON-CONFIG / U-DAEMON-SHUTDOWN），每个 unit 的判据使用系统服务标配概念（对照 systemd / launchd / Windows Service）。
4. **失败模式独立成文**：详见 docs/v0.1-failure-modes.md；本文档每个 unit 的"已知风险"段仅列该 unit 特有风险，泛域风险（rate limit、磁盘满等）一律见 failure-modes 文件。
5. **性能基准固定（单一）**：所有 perf 类判据的参考硬件 = **Windows 11 + 8GB RAM + SSD + Node.js 24**（CLAUDE.md 所记主开发机）。macOS 测试结果在 v0.1 作为信息记录（非阻塞性），v0.2 补。
6. **U-LLM-SPIKE 是 U-L2-MEM 的硬前提**：U-LLM-SPIKE 未闭环（未验证至少一种 LLM API 的 client-side truncate 可行），U-L2-MEM 不开工；U-LLM-SPIKE 闭环失败则 L2 推 v0.2。

## daemon 退出 timeline（单一权威，所有相关 unit 引用此组数字）

```
0s    : SIGTERM / SERVICE_CONTROL_STOP / CTRL_CLOSE_EVENT 收到
0–8s  : drain in-flight 任务（停止接受新任务，等待当前 checkpoint 写入完成）
8–11s : flush 状态到 SQLite（任务行 / scratchpad / 锁 / 日志缓冲）
11–15s: cleanup（释放锁文件 / 关闭 IPC socket / 关闭 SQLite handle）
15s   : 强制 SIGKILL（Windows: TerminateProcess）
```

**范围**：PRODUCT.md §17.1 要求的 7 个 MCP 工具（`cairn.scratchpad.write/read/list`、`cairn.checkpoint.create/list`、`cairn.rewind.preview/to`）的基础可用形态。不包含 L1/L2 memory checkpoint、rsync backend、daemon 常驻等桌面主体特性。

**验证方法（regression）**：
- `cd packages/mcp-server && npm test`（`wedge-acceptance.test.ts` 8 个 acceptance + 1 个 stdio smoke）
- `cd packages/daemon && npm test`（67 个 storage 单元测试）

**判据**（每条二值）：
- 7 个工具均可被 Claude Code 识别并调用（`cairn-wedge` 在工具列表中出现）✓/✗
- `scratchpad.write` 持久化写入 SQLite，`scratchpad.read` 能读回完全一致的值 ✓/✗
- `scratchpad.list` 返回所有 key + `updated_at` unix ms + `updated_at_iso` ISO 字符串 ✓/✗
- `checkpoint.create` 返回 ULID id + `git_head` 40 字符 hex（非 git 仓库返回 null） ✓/✗
- `checkpoint.list` 按 `created_at DESC` 返回，最新 checkpoint 在第一位 ✓/✗
- `rewind.preview` 返回 `files` 数组，每个文件路径为仓库相对路径 ✓/✗
- `rewind.to` 返回 `{ ok: true, restored_files, auto_checkpoint_id }`，文件内容字节级还原 ✓/✗
- daemon + mcp-server 全套测试通过（当前总计 ≥ 76 个 tests），`tsc --noEmit` 零错误 ✓/✗

**已知风险**：
- `checkpoints.label` 字段包含 `::stash:<sha>` 技术债，U-MIG-004 才清理（DESIGN_STORAGE.md §17.1 P2 清理项）
- `size_bytes` 总是 0（git-stash backend 不计算目录大小），U-RSYNC 后修正
- 楔工具只活在父 Claude Code 进程，subagent 不可直接访问（friction #1，长期靠 daemon IPC 解决）

### U-DONE-2：clean-tree rewind via git_head fallback

**范围**：当 checkpoint 时工作树 clean（无 dirty 文件），`rewind.to` 和 `rewind.preview` 使用 `git checkout HEAD -- <files>` 路径兜底，而非报错 "no stash backend recorded"。friction #2 长期方案 b。

**依赖**：U-DONE-1。

**验证方法（regression）**：
- `wedge-acceptance.test.ts` 中 "wedge — clean-tree checkpoint + rewind via git_head fallback" describe 块（8 个 it）

**判据**：
- clean-tree `checkpoint.create` 返回 `warning` 包含 "Working tree was clean"、"Rewind will restore the tree to git_head"、"gitignored files ... are left alone"，不包含 "cannot restore" ✓/✗
- dirty-tree `checkpoint.create` 返回结果中无 `warning` 字段 ✓/✗
- clean-tree `rewind.preview` 返回 `mode: 'git_head_clean'`，`files` 列出 checkpoint 后修改的所有文件 ✓/✗
- clean-tree `rewind.to` 还原 tracked 文件内容到 git_head 状态，新增 untracked 文件被删除 ✓/✗
- clean-tree `rewind.to` 在 HEAD 已移动后返回 `{ ok: false, error: /HEAD has moved/i }` ✓/✗
- clean-tree `rewind.to` 带 `paths` 参数只还原指定文件，其余文件不动 ✓/✗

**已知风险**：clean-tree 模式不还原 gitignored 文件（如 `node_modules`、`.env`），这是设计决策，UI 需明示。

### U-DONE-3：rewind paths 参数（per-file 粒度）

**范围**：`rewind.preview` 和 `rewind.to` 接受可选 `paths` 参数，限定 rewind 作用域到指定文件列表。friction #10 长期方案 a。不在 `paths` 内的 checkpoint 文件被列入 `skipped`，不出错。

**依赖**：U-DONE-1。

**验证方法（regression）**：
- `wedge-acceptance.test.ts` 中 "wedge — rewind paths param (friction #10 close)" describe 块（4 个 it）

**判据**：
- `rewind.to` 带 `paths: ['a.txt', 'c.txt']` 只还原 a.txt 和 c.txt，b.txt 保持现状 ✓/✗
- `rewind.to` 返回 `restored_files` 仅包含实际还原的文件，`skipped` 包含 `paths` 中未被 checkpoint 捕获的文件名 ✓/✗
- `rewind.preview` 带 `paths` 返回过滤后的 `files` 子集 + `skipped` 列表 ✓/✗
- `rewind.to` 带 `paths: []` 返回 `{ ok: false, error: /empty/i /omit/i }` ✓/✗
- 不带 `paths` 的 `rewind.to` 仍执行全范围还原（向后兼容） ✓/✗

**已知风险**：`paths` 过滤仅在 stash 模式和 git_head_clean 模式各自实现；U-RSYNC 落地后需补充 rsync backend 的等价 filtered 路径。

### U-DONE-4：write-effecting 工具的 auto-checkpoint（undo-undo 流程）

**范围**：`scratchpad.write` 和 `rewind.to` 在执行前自动创建 timeline 节点（`auto:before-*` 标签的 checkpoint），使用户可以撤销撤销（undo-undo）。提供 `skip_auto_checkpoint` 参数避免高频写污染时间线。

**依赖**：U-DONE-1。

**验证方法（regression）**：
- `wedge-acceptance.test.ts` 中 "wedge — auto-checkpoint on write-effecting tools" describe 块（5 个 it）

**判据**：
- `scratchpad.write` 在非 `skip_auto_checkpoint` 时，`checkpoint.list` 中出现 `auto:before-scratchpad.write:<key>` 标签的 checkpoint ✓/✗
- `scratchpad.write` 返回 `auto_checkpoint_id` 为 ULID 格式 id（dirty 工作树时），或 null（non-git 目录时不报错） ✓/✗
- `scratchpad.write` 带 `skip_auto_checkpoint: true` 时 `auto_checkpoint_id === null`，timeline 中无对应 auto 节点 ✓/✗
- `rewind.to` 后立即调 `rewind.to(auto_checkpoint_id)` 能还原到 rewind 前的状态（undo-undo 完整流程） ✓/✗
- non-git 目录中 `scratchpad.write` 成功，`auto_checkpoint_id` 字段存在于返回值（null 或 id） ✓/✗

**已知风险**：auto-checkpoint 不能 throw（tryAutoCheckpoint 有 try/catch），在 daemon 未启动或非 git 环境下静默返回 null，主操作不受影响。

### U-REWIND-TASK-SCOPE：rewind.to 新增 task_id 入参语义

**范围**：`rewind.to` 工具新增可选 `task_id` 入参——当传 `task_id=` 时，自动从该 task 名下的所有 checkpoint 收集涉及的文件路径，作为 paths 过滤器使用。无 `task_id` 时维持当前行为（全文件 rewind 或显式 paths 过滤，v0.0 兼容）。覆盖 AC for US-2 第 2 条（3 条任务之间独立 rewind）。

**依赖**：U-DONE-3（paths 过滤），U-DONE-5（task_id slicing）。

**验证方法**：`acceptance-test`

**判据**：
- 创建 task A 下 2 个 checkpoint，task B 下 1 个 checkpoint，分别涉及不重叠的文件集 ✓/✗
- `rewind.to(checkpoint_in_A, task_id=A)` 仅还原 task A 文件集，task B 文件不受影响 ✓/✗

### U-DAEMON-IPC：客户端协议（NDJSON over Unix socket / Windows named pipe）+ 连接管理

**依赖**：U-DAEMON-PROC（进程已起、锁已持有）。

**验证方法**：
- `acceptance-test`：spawn daemon → 建 IPC 连接 → 发 ping 请求（`{ "method": "ping" }`）→ 断言收到 pong 响应（`{ "result": "pong", "protocol_version": 1 }`），往返延迟 < 50ms

- IPC 服务端在 daemon 启动 1s 内开始监听 ✓/✗
- ping/pong 往返延迟 p99 < 50ms（本机 socket 级别，基准硬件：Windows 11 + 8GB RAM + SSD + Node.js 24） ✓/✗
- 客户端断连后 daemon 继续运行（`kill -0 <pid>` 成功），下一个客户端可以重新连接 ✓/✗
- 协议版本不匹配时（客户端 `protocol_version: 999`），daemon 返回 `{ "error": "protocol version mismatch" }` 而非崩溃 ✓/✗

**IPC 协议演进规则（v3 补）**：
- 客户端首次连接发送 `protocol_version`；daemon 回复支持版本范围 `[min, max]`
- 新客户端 + 旧 daemon：客户端 fallback 到 `(max - 1)` 协议；若 `max < client 最低支持版本`，明确报错"client 与 daemon 版本不匹配，请升级 daemon"
- 旧客户端 + 新 daemon：daemon 接受 `(current_version - 1)` 协议直到 v0.2 强制升级；记录"使用旧协议连接"日志
- 兼容窗口：v0.1 daemon 必须接受 `protocol_version=1`；v0.2 daemon 接受 `protocol_version IN (1,2)`；v0.3 起强制 `protocol_version >= 2`

**已知风险**：Windows 上 Unix domain socket 在 Node 24 已可用，但需在 U-CROSS-PLAT 中专门验证（`\\.\pipe\` 和 `/tmp/xxx.sock` 两路径均需覆盖）。

### U-DAEMON-SHUTDOWN：优雅退出（drain 进行中任务 + 持久化状态 + 释放锁）

**范围**：daemon 收到 SIGTERM / SIGINT 后执行优雅退出序列：停止接受新任务 → drain 进行中的子 Agent（等待当前 checkpoint 写入完成）→ flush 任务状态到 SQLite → 释放 advisory lock → 删除 PID 文件 → 退出 exit 0。对应 systemd 的 TimeoutStopSec 和 KillMode=process 概念。

**依赖**：U-DAEMON-PERSIST（状态 flush），U-DAEMON-SUBAGENT-KILL（子 Agent 终止），U-DAEMON-SUBAGENT-IPC-BRIDGE（bridge 关闭）。

**退出 timeline**（引用 §0 daemon 退出 timeline，单一权威）：
- 0–8s drain / 8–11s flush SQLite / 11–15s cleanup / 15s 强制 SIGKILL

**验证方法**：
- `acceptance-test`：启动 daemon + 创建进行中任务 → 发 SIGTERM → 等待退出 → 检查 SQLite 中任务状态已正确持久化，PID 文件已删除

**判据**：
- 发 SIGTERM 后 daemon 在 15s 内退出（drain=8s + flush=3s + cleanup=4s，超时强制 SIGKILL）✓/✗
- drain 阶段（0–8s）：不接受新任务（新 IPC 请求返回 503 Shutting Down），在途任务等待当前 checkpoint 写入完成 ✓/✗
- flush 阶段（8–11s）：所有进行中任务状态写入 SQLite（status='PAUSED'），日志缓冲 flush ✓/✗
- cleanup 阶段（11–15s）：IPC socket 文件删除，PID 文件删除，SQLite handle 关闭 ✓/✗
- 退出后 `~/.cairn/daemon.pid` 文件不存在 ✓/✗
- 退出后 SQLite 中进行中任务的 status 为 'PAUSED' 或 'RUNNING'（不为 'FAILED' 或空）✓/✗
- 退出后 IPC socket 文件不残留 ✓/✗
- 正常退出时 exit code = 0；超时 SIGKILL 时 exit code = 130 ✓/✗

**Windows 特异（v3 补）**：
- `SERVICE_CONTROL_STOP`（SCM）或 `CTRL_CLOSE_EVENT`（控制台关闭）触发同上 timeline
- Windows `SetConsoleCtrlHandler` 默认 5s 超时；daemon 在 4.5s 内主动 flush 完成（或主动 SIGKILL 自己），避免被 OS 强杀

**已知风险**：drain 阶段（0–8s）内若有子 Agent 仍在执行文件 I/O，强制退出可能产生不完整写入，需 U-CORRUPT 兜底。

### U-PERF：基础性能预算（每次 release candidate 触发一次）

**基准硬件**：**Windows 11 + 8GB RAM + SSD + Node.js 24**（CLAUDE.md 所记主开发机，单一基线）。macOS 性能数据作为信息记录，不纳入 ship 闸。

**范围**：为以下操作建立 p50 / p99 / p999 基线，防止性能随 Unit 积累而悄悄退化。

**依赖**：U-RSYNC，U-L1-MEM（L1 快照增加了 checkpoint 的工作量）。

**验证方法**：
- `regression`：`packages/daemon/tests/perf/benchmark.ts`，用 `performance.now()` 计时，重复 100 次取 p50 / p99 / p999

**判据**（在基准硬件上，工作树 1000 个文件，总大小 10MB，全量 snapshot 场景）：
- `checkpoint.create`（含 rsync 快照）：p50 < 2s，p99 < 5s，p999 < 10s ✓/✗
- `rewind.to`（含文件还原，同等规模）：p50 < 2s，p99 < 5s，p999 < 10s ✓/✗
- `scratchpad.list`（1000 个 key，无 blob 分流）：p50 < 20ms，p99 < 100ms，p999 < 500ms ✓/✗
- IPC ping/pong 往返（基准硬件本机 socket）：p50 < 10ms，p99 < 50ms，p999 < 200ms ✓/✗
- 首次提示渲染（UI 收到 daemon IPC 响应到对话气泡出现）p99 ≤ 200ms（基准硬件，本机 IPC 路径，不含 LLM 延迟）✓/✗
- 性能回归阈值：单次超过上次 p99 的 **3 倍**触发 CI 失败（不是警告，是硬失败）

**证据产物**：每次 release candidate 的 p50 / p99 / p999 对比表记录在 `docs/perf-baseline.md`。

**已知风险**：rsync `--link-dest` 增量策略可大幅降低后续 checkpoint 时间；基线应在第一次（全量）和第二次（增量）分别测量，上述阈值适用于第一次全量场景。

## 8. 失败回退路径

### 触发条件 1：L2 LLM API 兼容性 spike 失败

**条件**：U-LLM-SPIKE 发现主要 LLM API 在 cairn client-side conversation store 方案下无法可靠 truncate。

**行动**：
1. 在 `docs/llm-truncate-spike-result.md` 中记录失败结论
2. U-L2-MEM 状态改为"失败回退"，不纳入 v0.1 实施
3. PRODUCT.md §4.4 AC for US-4 第 3 条降级为"UI 明示文件与 git 已回到 t1；注意：对话记录未回滚（v0.2 覆盖）"
4. v0.1 ship 时间从 W15 提前到 W14（w4-path-decision.md §5 风险 3）
5. U-E2E-US4 的 L2 判据改为"UI 明示 L2 未回滚（v0.2 覆盖）"，仍可完成验证

### 触发条件 2：daemon IPC 设计阶段卡死

**条件**：U-DAEMON-IPC 实施阶段发现 NDJSON over socket 在 Windows 上存在不可解问题，或 p99 > 250ms（超过阈值 5 倍），无法在合理时间内修复。

**行动**：
1. 评估备选 IPC 方案：HTTP localhost 或 Windows named pipe 方案
2. 若备选方案 ≤ 1 天可验证，切换方案并重新跑 U-DAEMON-IPC 验证
3. 若备选方案评估周期 > 3 天，降级为：桌面主体推迟，仅做 CLI 形态（无 Tauri UI）的 v0.1；PRODUCT.md F-2（宠物外观）推 v0.2
