## 1. 产品决议（这是为什么要锁这个文件）

Inspector 在 v0.2 不做 Jira-style status dashboard，做 **Live Run Log**：一个 read-only、event-oriented 的 host-level coordination kernel 窗口。它呈现 agents / subagents / tasks / blockers / outcomes / checkpoints / conflicts 随时间流动的事件，让 human supervisor 能复盘和监督多 agent 协作现场。

这是产品形态决议，**不是 UI 优化决议**。误把它当成"Inspector UI 美化任务"会丢失下面所有边界判断。

**Cairn 的产品 framing**：host-level multi-agent coordination kernel。kernel 的 UI 形态从来不是 dashboard，是 *window into itself*——Linux `/proc`、dtrace、journalctl 都是事件流。Run Log 是 Cairn 的 `/proc` 层，不是它的产品本身。

**典型形态**（草稿，非锁定 UI 规格）：

```text
Cairn
3 agents active · 2 tasks running · 1 blocked · 1 waiting review

[16:42:10] codex-a started task T-182 "refactor outcomes evaluator"
           checkpoint ckp_91 created before edits
[16:43:02] subagent-sonnet wrote scratchpad result
           "parser edge cases reviewed; 2 failing fixtures found"
[16:45:01] task T-182 BLOCKED
           question: "Should tests_pass run package root or workspace root?"
[16:49:30] user answered blocker
[16:50:11] codex-b resumed task T-182 from resume_packet (ckp_91)
[16:55:22] outcome evaluate FAIL — tests_pass: 2 tests red
[17:03:44] outcome evaluate PASS — task T-182 DONE
```

每行带明确动词；事件按时间倒序或正序（待 W2 决定）；不显示百分比、不显示进度条。

## 2. Non-Goals（v0.2 Inspector **不**是什么）

下列设计如果在 v0.2 PR / issue / spec 里出现，按本节直接 veto：

- ❌ **Jira / Linear / Asana 类 task board** — Cairn 不是项目管理工具（PRODUCT.md §1.3 反定义）
- ❌ **Progress bars / 完成度百分比** — Cairn 不知道一个 task "完成多少 %"，也不假装知道
- ❌ **Observability SaaS / APM 形态**（Datadog / Honeycomb 风格仪表盘）— Cairn 不是观测工具，Run Log 是 kernel 自带的 `/proc` 窗口
- ❌ **Desktop 直接写 state**（answer blocker / resolve conflict / rewind 等 mutation 路径）— v0.2 Inspector 严格 read-only；mutation 推迟到 v0.3 daemon API
- ❌ **Agent 化身 / 拟人 chat UI**（v1 桌面宠物路线）— PRODUCT.md §8.2 与 v1 区别表已 veto
- ❌ **进度 / 截止时间 / velocity / burndown** — Cairn 不追踪 deadline，不评估 agent 工作量
- ❌ **NL query 框 / "ask Cairn what happened"** — v0.2 不用 LLM 做 Run Log 摘要，保持确定性

## 3. Scope（按 v0.2 周次划分）

### W1 — Events 表（持久层 prerequisite）

**目标**：Run Log 不再依赖"从 N 张表 created_at/updated_at 推时间线"，要有正式 append-only 事件源。

- migration `011-events`：`cairn_events` 表，append-only，字段建议 `id / occurred_at / actor_kind / actor_id / event_type / target_kind / target_id / payload_json / severity`
- repo 层（`packages/daemon/src/storage/repositories/*.ts`）所有写路径同步 emit 事件；测试覆盖到"每次 state transition 必有对应 event"
- 不上完整 event sourcing 重构：现有 8 类 state object 表保留为 source of truth，events 是观察层
- severity 三档：`INFO / NOTICE / ALERT`（heartbeat 默认 INFO；state transition NOTICE；blocker open / conflict / outcome FAIL ALERT）

**为什么 W1 必须做这个而不是直接做 UI**：W1 之前 Run Log 只能从 created/updated 聚合，看不到中间态（`RUNNING → BLOCKED → RUNNING` 只剩最后 `updated_at`），叙事感会假。在 events 表落地前做 UI = 把 UI 绑在伪时间线上。

### W2 — Desktop Read-Only Live Log

- desktop-shell 加 Live Log 视图，作为 PRODUCT.md §8.2 "click → Inspector panel" 那一档的主视图
- 数据通道 v0.2 起步形态：desktop 进程 readonly 打开同一 SQLite，poll `MAX(rowid) FROM cairn_events`（WAL 模式多 reader 安全，零新基础设施）
- 渲染：时间线 + 每行带明确动词（"started" / "blocked" / "resumed" / "evaluated PASS"）
- 顶部状态条：active agents / running tasks / blocked / waiting review / failed outcomes 五个数字（聚合自现有表）
- **不做**：filter / folding / 详情抽屉（W3）；新 daemon endpoint（v0.3）

### W3 — Filters / Folding / 详情抽屉

- 按 task / agent / event_type / severity 过滤
- 重要事件白名单（state transition / blocker / outcome / conflict / rewind）默认显示；heartbeat / scratchpad write 等噪声事件折叠
- 点击事件 → 详情抽屉：原始对象 + 关联 scratchpad / outcome evidence / resume_packet（拼自现有 read-only repo）

### v0.3（**不在 v0.2 scope，仅锁方向**）

- daemon HTTP / Unix socket / Windows named pipe API（暴露 events stream + read APIs）
- Action Panel：answer blocker / resolve conflict / rewind / open resume packet
- desktop 作为 supervisor 身份的 auth / agent_id 模型（desktop 不是 agent，是 supervisor，需要新身份模型）

## 4. 显式判定边界（争论时按这条裁决）

| 提案 | v0.2 接受？ | 理由 |
|---|---|---|
| 给每个 task 加进度条 | ❌ | progress bar 假装知道完成度，Cairn 没有这个语义 |
| 顶部加 "today's velocity" 图表 | ❌ | velocity = 项目管理度量，not kernel concern |
| 桌面右键 task → mark done | ❌ | mutation 路径，v0.3 才考虑，且要走 agent 接口 |
| Live Log 按 severity 染色 | ✅ | INFO/NOTICE/ALERT 已在 W1 events 表里建模 |
| conflict 高亮成红色 + 点击进 arbitration | ✅ 高亮；❌ desktop 直接 arbitrate | 高亮 ok；arbitration 是 mutation，v0.3 |
| "export run log to JSON" 按钮 | ✅（W3 nice-to-have） | read-only 操作，对齐"本地优先"原则 |
| 加 LLM "ask what happened today" | ❌ | NL summarization 引入非确定性，v0.2 保持确定性渲染 |

## 5. 风险

- **R1：events 表 schema 一锁就难改**。缓解：W1 之前先做一轮 schema review，覆盖 8 类 state object 的所有写路径，确认 `payload_json` 不被滥用成大杂烩
- **R2：Run Log 噪音淹没真信号**。缓解：W3 之前 default view 只显示 ALERT + NOTICE，INFO 折叠
- **R3：被外部使用者解读为 observability 产品**。缓解：本 plan §1 + §2 + 文档措辞统一用 "kernel window"，绝不用 "observability"
- **R4：v0.2 范围里又被加进 mutation 需求**。缓解：本 plan §3 W2 / §4 表格里硬钉 read-only
