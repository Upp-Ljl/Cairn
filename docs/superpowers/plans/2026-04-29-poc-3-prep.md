---

## 0. 文档定位

### 0.1 这是什么

本文是 PoC-3 的执行包。使用方式：

1. 用自己的 Claude Sonnet（coding plan）把 §3 的 20 条 NL prompt 逐条 paste 进去跑；
2. 每条返回结果按 §4 的 5 维度打分（0-10），填入 §5 的结果表格；
3. 看完数字，查 §6 决策矩阵，得出 verdict；
4. 把 verdict 文本填回 `ARCHITECTURE.md` ADR-4 的 🚧 锚点。

### 0.2 与 pre-impl-validation §3.3 的关系

`pre-impl-validation.md` §3.3 原版要求 Sonnet vs Llama 3.1 8B vs Qwen 2.5 7B 三模型对比，每条 20 × 3 = 60 个打分点。**本文是单边简化版**：只跑 Sonnet，不跑本地 7B。理由：v0.1 已在 ADR-4 默认用 Sonnet API，本地 7B 对比的价值在决定"是否引入本地 fallback"，这是 v0.2 的决策，不阻塞 v0.1。v0.2 启动前再补本地 7B 对比，决定是否做双引擎。

原版的 300 个打分点因此缩减到 20 × 5 = 100 个打分点。

### 0.3 与 ARCHITECTURE.md ADR-4 的关系

跑完 §5 得出整体均分后，按 §6 决策矩阵找到对应 verdict 文本，复制到 `ARCHITECTURE.md` ADR-4 段的 🚧 PoC-3 锚点，完成回填。回填后该锚点从 🚧 变为 ✓。

## 1. 执行步骤（用户视角）

**Step 1：准备 Sonnet 会话**

打开一个新的 Claude Sonnet coding plan 会话，把 §2 的 system instruction 整段 paste 进去（或设为 system prompt）。这只需要做一次，后续 20 条 NL 用例都在同一个会话里跑。

**Step 2：逐条 paste NL prompt，记录输出**

把 §3 里每条用例的「NL prompt」完整 paste 给 Sonnet，等它返回结构化 JSON 输出。每条预计 1-3 分钟（含阅读输出 + 打分）。20 条全部跑完约 30-60 分钟。

**Step 3：打分并填表**

按 §4 的 5 维度 rubric，对每条输出打分（0-10），填入 §5 的结果表格。参考每条用例里的「打分参考」做校准。

**Step 4：对照决策矩阵，写 verdict，回填 ADR-4**

§5 表格填完后，算出整体均分和各类均分，查 §6 决策矩阵，把对应的 ADR-4 回填文本写入 `ARCHITECTURE.md`。

## 2. Sonnet 角色提示（每次新会话开始时 paste）

把以下 system instruction 完整 paste 给 Sonnet，建立角色后再开始跑 NL 用例：

```
你正在扮演 Cairn 的 Dispatch 模块。Cairn 是本机多 agent 协作的内核（Agent OS），
它不写代码，只把用户的自然语言需求翻译成 agent 能直接使用的 prompt。

每收到一条用户 NL 需求，你必须输出一个结构化 JSON：

{
  "intent": "简短意图标签，如 dispatch.refactor / dispatch.fix / dispatch.review / inspect.history / dispatch.newfeature",
  "agent_choice": "Claude Code | Cursor | Aider | None（说明原因）",
  "prompt_to_agent": "你要派给该 agent 的 prompt 全文（必须可以直接发给 agent 执行，不加废话）",
  "history_keywords": ["关键词1", "关键词2", "...（用于从 scratchpad / checkpoint 检索相关历史）"],
  "risks": ["需要 preview 的危险操作", "需要二次确认的不可逆动作", "...（如果没有风险可以是空数组）"],
  "uncertainty": "你拿不准的地方（如果没有则写 null）"
}

上下文假设：
- 当前活跃的 agent 是 Claude Code（已有当前 repo 的上下文），备选 Cursor。
- Cairn 已记录的 scratchpad / checkpoint 里有过去 1-7 天的开发历史（含需求记录、中间决策、文件改动记录）。
- 你不直接执行，只输出结构化派单决定，用户审查后才会转发给 agent。
- 如果用户 NL 模糊或者需要查历史才能解析，在 uncertainty 字段里明确说明。
- 严格按 JSON 格式输出，不加任何解释文字。
```

## 3. 20 条 NL 测试用例

分 4 类：A（简单派单）/ B（复合派单）/ C（模糊意图）/ D（危险/边界 case）。

---

### 类 A：简单派单（5 条）

预期高分区。单一 agent，动词明确，目标清晰，没有歧义。

---

**A.1**

**NL prompt**：「把整个 repo 里 utils_v2 这个命名改成 string_helpers，含 import / 注释。改完跑一遍测试，挂了停下来问我。」

**预期 intent**：`dispatch.refactor`（全 repo 重命名 + 测试守卫）
**预期 agent**：Claude Code（多文件 batch refactor 是 CC 强项；Cursor 更适合单文件）
**关键约束 / 风险**：测试失败必须停下来（不能自动继续）；涉及 import 路径需确认无遗漏
**打分参考**：
- 维度 1（意图正确）：识别"全 repo 重命名 + 跑测试 + 失败停下"三项全给 9-10；识别其中两项给 5-7；只识别"重命名"给 2-3
- 维度 2（agent 选对）：选 Claude Code 给 10；选 Cursor 给 5（能做但不是最优）；选 None 给 0
- 维度 3（prompt 合理）：含"跑测试 + 测试挂了停下问"约束给 9-10；只含"重命名"给 4；prompt 里没有约束给 2
- 维度 4（历史关键词）：含 `utils_v2` / `string_helpers` 给 9-10；只有一个给 5；关键词都不对给 2
- 维度 5（风险提示）：识别"测试挂了不继续"是需要 agent 执行前确认的约束给 8-10；空风险给 3；编造不存在的风险给 2

---

**A.2**

**NL prompt**：「帮我写一个函数 formatBytes(n: number): string，把字节数转成人类可读字符串（例：1024 → '1 KB'），放在 src/utils/format.ts，然后写对应的单测。」

**预期 intent**：`dispatch.newfeature`（新增函数 + 单测）
**预期 agent**：Claude Code
**关键约束 / 风险**：文件路径明确（`src/utils/format.ts`）；需要写单测；函数签名明确
**打分参考**：
- 维度 1：识别"新函数 + 单测 + 指定路径"三项给 9-10；识别两项给 6-7；只说"写个函数"给 3
- 维度 2：Claude Code 给 10；Cursor 给 6；None 给 0
- 维度 3：prompt 含函数签名 + 路径 + 单测要求给 9-10；缺任何一项给降 2 分
- 维度 4：含 `formatBytes` / `src/utils/format.ts` / `单测` 给 9-10；关键词零相关给 2
- 维度 5：这条没有风险，risks 为空数组或 null 给 10；编造风险给 3

---

**A.3**

**NL prompt**：「review 一下 packages/mcp-server/src/index.ts 这个文件，找出潜在的类型不安全点和 error handling 遗漏，给我一份 review 意见，不要改代码。」

**预期 intent**：`dispatch.review`（只读 review，不改动）
**预期 agent**：Claude Code（或 Cursor，两者均可，重点在"只读"约束）
**关键约束 / 风险**：严格只读，不改任何代码；输出是 review 意见而非直接 fix
**打分参考**：
- 维度 1：识别"code review + 不改代码 + 输出意见"给 9-10；漏掉"不改代码"约束给 5；误解成"修代码"给 1
- 维度 2：Claude Code 或 Cursor 均给 10；None 给 0
- 维度 3：prompt 含"只看不改 + 指定文件路径 + 输出类型（意见列表）"给 9-10；缺"只读"给 5
- 维度 4：含文件路径 / `mcp-server` / `error handling` 给 9-10
- 维度 5：无风险（只读操作）；risks 为空给 10；识别"只读"本身不是风险给 8；编造风险给 2

---

**A.4**

**NL prompt**：「在 packages/daemon/src/storage/db.ts 顶部加一行注释：// Core database module – do not modify without migration。不用跑测试。」

**预期 intent**：`dispatch.edit`（微小编辑，无测试需求）
**预期 agent**：Claude Code（或 Cursor，小改动两者都合适）
**关键约束 / 风险**：改动极小（一行注释），不需要跑测试；路径明确
**打分参考**：
- 维度 1：识别"加注释 + 不跑测试 + 路径明确"给 9-10；误解成"做大改动"给 1
- 维度 2：Claude Code 或 Cursor 均给 10
- 维度 3：prompt 含精确注释内容 + 文件路径给 10；只写"加注释"给 5
- 维度 4：含 `db.ts` / `packages/daemon` 给 9-10
- 维度 5：无风险；risks 为空或 null 给 10

---

**A.5**

**NL prompt**：「把 src/config.ts 里的 DEFAULT_TIMEOUT 从 5000 改成 10000，然后搜一下整个 repo 有没有 hardcode 了 5000 这个数字的地方，有的话列出来但不要改。」

**预期 intent**：`dispatch.edit`（改常量 + 只读搜索）
**预期 agent**：Claude Code（改文件 + 全 repo 搜索两步合并）
**关键约束 / 风险**：改动 config 有连带影响；搜索结果只列出不改；需要确认无其他 hardcode
**打分参考**：
- 维度 1：识别"改常量 + 全 repo 搜索 + 搜索只读"三项给 9-10；漏掉"搜索只读"给 6
- 维度 2：Claude Code 给 10；Cursor 给 6
- 维度 3：prompt 含精确改动（5000→10000）+ 搜索范围 + "列出不改"约束给 9-10
- 维度 4：含 `DEFAULT_TIMEOUT` / `src/config.ts` / `5000` 给 9-10
- 维度 5：识别"改超时参数可能影响下游行为"是 risk 给 8-10；空 risks 给 5（此条有潜在风险）

### 类 B：复合派单（5 条）

中等难度。多个动作串联，可能需要拆分或排序。

---

**B.1**

**NL prompt**：「先把 auth 模块的测试跑一遍看看哪些挂了，挂了就让 Claude Code 修，修完再跑一遍确认。如果两次都挂同一个测试，停下来告诉我，不要继续。」

**预期 intent**：`dispatch.fix`（测试先行，修 + 验证，循环保护）
**预期 agent**：Claude Code
**关键约束 / 风险**：多步骤（跑测试 → 修 → 再跑）；同一测试挂两次不继续（防止无限循环）；auth 改动有安全风险
**打分参考**：
- 维度 1：识别"run tests → fix → re-run → halt if same failure twice"给 9-10；漏掉"第二次挂停下"给 6；只识别"修测试"给 3
- 维度 2：Claude Code 给 10
- 维度 3：prompt 含三步流程 + 停止条件给 9-10；缺停止条件给 5
- 维度 4：含 `auth` / `test` / `failing` 给 9-10
- 维度 5：识别"auth 模块改动有安全影响"或"无限修复循环"给 8-10；空 risks 给 3

---

**B.2**

**NL prompt**：「给 cairn 的 scratchpad.write 工具加一个参数校验：key 不能包含 / 开头和结尾的斜杠，否则报错。改完写测试，测试绿了再 commit，commit message 用 feat(scratchpad): validate key format。」

**预期 intent**：`dispatch.feature`（参数校验 + 测试 + commit 规范）

**打分参考**：
- 维度 1：识别"加校验 + 写测试 + 指定 commit message"给 9-10；漏掉 commit 格式要求给 6
- 维度 2：Claude Code 给 10
- 维度 3：prompt 含校验逻辑 + 测试 + 精确 commit message 格式给 9-10
- 维度 4：含 `scratchpad.write` / `key validation` / `commit` 给 9-10
- 维度 5：识别"MCP 工具接口变动可能是 breaking change（如果 agent 传了 /开头 key）"给 8-10；空 risks 给 4

**B.3**

**NL prompt**：「帮我做一个 end-to-end 检查：1）列出 packages/daemon/src/storage/migrations/ 下所有 migration 文件；2）看看 migrations/index.ts 里的 ALL_MIGRATIONS 数组顺序是否和文件名的数字前缀一致；3）如果不一致，告诉我哪条不对，不要自己改。」

**关键约束 / 风险**：纯只读；输出结论不改代码；多步骤有顺序依赖
**打分参考**：
- 维度 1：识别"列文件 → 对比数组顺序 → 报告不一致 + 只读"给 9-10；漏掉"只读"约束给 5
- 维度 2：Claude Code 给 10；None + 说明"用户自己看"给 6；Cursor 给 6
- 维度 3：prompt 含三步顺序 + "只报告不改"约束给 9-10
- 维度 4：含 `migrations` / `ALL_MIGRATIONS` / `index.ts` 给 9-10
- 维度 5：无风险（只读）；risks 为空给 10；识别"只读确认"给 8

**B.4**

**NL prompt**：「把 mcp-server 的依赖都升到最新（npm update），升完跑全套测试，测试挂了就回滚（npm install 用旧的 package-lock），然后把哪个依赖升级导致测试挂告诉我。」

**B.5**

### 类 C：模糊意图（5 条）

难度高。需要先查历史或澄清才能解析，Sonnet 应该在 uncertainty 字段里说清楚它拿不准的地方。

---

**C.1**

**NL prompt**：「auth 改的方向不对，和上周五我说的需求不一致，帮我找到对的 agent 重做这一块。」

**预期 intent**：`dispatch.redirect`（方向纠错 + 需要查历史需求）
**预期 agent**：Claude Code（已有 auth 模块上下文）
**关键约束 / 风险**：必须先查 scratchpad / checkpoint 历史找到"上周五说的需求"；不能直接开始改；需要先展示找到的历史需求让用户确认
**打分参考**：
- 维度 1：识别"方向纠错，不是功能新增"并说明"需要先查上周五历史"给 9-10；直接生成 prompt 不查历史给 3；空 intent 给 0
- 维度 2：Claude Code 给 9-10；说明"需要先查历史再确定 agent"给 8；直接选 None 给 3
- 维度 3：prompt 含"参考 scratchpad 历史 + 请用户确认历史需求再开始"给 9-10；直接改 auth 给 2
- 维度 4：含 `auth` / `上周五` / `需求` / `history` 给 9-10
- 维度 5：识别"不查历史就改方向是高风险"给 9-10；空 risks 给 2

---

**C.2**

**NL prompt**：「上次那个 migration 还没测完，继续。」

**预期 intent**：`inspect.resume`（恢复上次中断的任务，必须查历史才知道是哪个 migration）
**预期 agent**：无法确定，需要先查（Sonnet 应在 uncertainty 里说明）
**关键约束 / 风险**：极度依赖历史上下文；不知道是哪个 migration；不知道"还没测完"的状态是什么
**打分参考**：
- 维度 1：识别"需要查 checkpoint / scratchpad 找最近的 migration 任务"给 9-10；直接说"继续 migration X"（没有历史依据）给 2
- 维度 2：agent_choice 含"需要先查历史再确定"给 9-10；直接选 CC 不说明给 4
- 维度 3：prompt 含"先用 cairn 查最近 migration 任务，找到后再继续测试"给 9-10；直接生成无依据的 prompt 给 2
- 维度 4：含 `migration` / `test` / `checkpoint` 给 9-10
- 维度 5：识别"不查历史就继续可能操作了错误的 migration"给 9-10；空 risks 给 2

---

**C.3**

**NL prompt**：「昨天的 schema 设计和我脑子里想的不一样，改回去。」

**预期 intent**：不明确——可能是 rewind / 可能是 Dispatch 新的 edit 任务（取决于"脑子里想的"是什么）
**预期 agent**：Claude Code（如果 dispatch edit）；或无 agent（如果 rewind，Cairn 自己处理）
**关键约束 / 风险**：「改回去」可能是 rewind（危险，破坏性操作）也可能是新的 edit；必须先弄清楚"脑子里想的"是什么；不能静默推断
**打分参考**：
- 维度 1：识别"意图不明——rewind 还是 edit？需要澄清"给 9-10；直接当 rewind 给 2；直接当 edit 给 3
- 维度 2：agent_choice 为 None 并解释"需要先澄清意图"给 9-10；直接选 CC 给 3
- 维度 3：prompt 含"请用户澄清：(a) 想回到昨天某个 checkpoint？还是 (b) 有新的 schema 设计要实现？"给 9-10；直接给改动 prompt 给 2
- 维度 4：含 `schema` / `昨天` / `checkpoint` 给 9-10
- 维度 5：识别"rewind 是不可逆操作，必须先 preview"给 9-10；空 risks 给 2

---

**C.4**

**NL prompt**：「subagent B 的结果和 subagent A 的 schema 对不上，把 schema 那边让 CC 修一遍。」

**预期 intent**：`dispatch.fix`（修 schema 让 B 的结果能对上），但需要从 scratchpad 查 A 和 B 的实际结果
**预期 agent**：Claude Code（修 schema）

**打分参考**：

**C.5**

**D.1**

## 4. 打分 Rubric（5 维度）

每个维度独立 0-10 分，不互相影响。

| 维度 | 0 分锚点 | 5 分锚点 | 10 分锚点 |
|---|---|---|---|
| **1. 意图正确** | 完全没读懂用户在说什么，或者意图标签和实际请求完全无关 | 抓住主要动作，但漏掉了次要约束（如"不要改代码"/"失败停下"等限制词） | 主动作 + 次要约束 + 边界条件全部识别，intent 标签精准 |
| **2. agent 选对** | 选了完全不合适的 agent（如推荐 None 去执行代码改动任务） | 选对了类别，但理由薄弱或没有理由，或选了次优 agent | 选对 + 给出选型理由（为什么这个 agent 比其他选项更合适） |
| **3. prompt 合理** | 生成的 prompt 模糊、矛盾，或包含错误信息，无法直接使用 | prompt 能用，但缺关键约束（如缺"测试失败停下"或缺目标文件路径） | prompt 直接 paste 给 agent 就能跑，约束完整，无需用户再补充 |
| **4. 历史关键词对** | 关键词与请求完全无关，或者全部为空（类 C 除外）；或者包含大量噪音词 | 抓住一两个相关词，但遗漏最关键的词（如只写"修改"不写具体文件名） | 关键词完整覆盖请求里提到的实体（文件路径、函数名、模块名、时间词）且无冗余 |
| **5. 风险提示** | 危险操作零提示；或完全无 risks 字段（类 A 的零风险场景除外） | 识别到存在风险，但描述过于模糊（如"可能出错"，没有具体风险点） | 准确识别每个具体风险 + 给出对应的 preview / 二次确认建议 |

**打分注意事项**：

- 类 A（简单派单）的风险维度：这类任务通常风险较低，如果 Sonnet 正确给出空 risks 数组，维度 5 给 8-10 分；如果它对零风险任务编造风险，给 2-3 分。
- 类 C（模糊意图）：意图不明时，Sonnet 应在 intent 或 uncertainty 里明确说"需要先查历史"，而不是直接猜一个意图生成 prompt。直接猜 intent 不说明不确定性，维度 1 最高给 4 分。
- 类 D（危险 case）：维度 5 权重在评判时可以重视，但打分数值仍是 0-10，不做特殊加权。

**单条 NL 总分**：5 维度的算术均值（= 五个维度分数之和 ÷ 5）。

**整体均分**：20 条单条总分的算术均值。

## 8. 失败时退路

**如果 Sonnet 输出格式不稳定（不返回 JSON）**

在 system instruction 末尾加：「严格按 JSON 格式输出，不加任何解释文字，不加 markdown 代码块标记。」重跑该条用例即可。

**如果某条 NL 你拿不准给几分**

在 §5 表格的"备注"列标"拿不准"，把该条 Sonnet 输出截图或粘贴到备注里，后续可以拿到 Cairn dev session 二次评定。一条拿不准不影响其他 19 条的计分。

**如果 20 条跑完后发现你的金标准有问题**

例如：你以为某条应该选 Claude Code，但实际上 Cursor 更合适——此时不需要重做整套。只需：
1. 修订那一条的"预期 agent"金标准；
2. 重新对那类（A/B/C/D）的相关条目打分；
3. 如果整体均分因此变化超过 0.5 分，重新查决策矩阵。

**如果整体均分在决策矩阵边界附近（6.5-7.5 之间）**

不要强行归类。在 ADR-4 回填文本里明确写"均分 X.X，接近 7.0 边界，v0.1 选择半模板化以降低风险，W4 中期 review 后可调整"。

## 9. 与 PoC-4 的关系

PoC-3 和 PoC-4 的数据完全独立，互不依赖。两个可以并行推进，互不阻塞。

- PoC-3 结论影响 ADR-4（LLM 选型）→ 影响 W5-W7 Dispatch v1 编码路径
- PoC-4 结论影响 §6.4（消息可达架构）→ 影响 W4-W5 / W7-W9 编码

两个都通过（PoC-3 均分 ≥ 7，PoC-4 调用率 ≥ 70%）→ W4-W7 编码按原 ARCHITECTURE.md 设计推进。任一不通过 → 各走各的失败应对路径（详见 `pre-impl-validation.md` §3.3 和 §3.4 的失败应对段）。
