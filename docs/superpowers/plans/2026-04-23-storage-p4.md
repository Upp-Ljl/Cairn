# Cairn Storage P4 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Light up the reserved `tasks.description_vec` column by wiring sqlite-vec into the daemon — adding an opt-in loader on `openDatabase`, a migration that creates the `vec_tasks` virtual table (plus a tiny bridge table that maps sqlite-vec's INTEGER rowid to the TEXT ULID `tasks.id`), and a `tasks-vec` repository that sets / clears / searches embeddings. P1–P3 tests remain green on platforms without the native sqlite-vec binary.

**Architecture:** sqlite-vec is a loadable SQLite extension; its NPM package (`sqlite-vec`) exposes a `load(db)` helper compatible with `better-sqlite3` v11. Loading is opt-in via `openDatabase(path, { vec: true })` — P1–P3 callers unchanged. Availability is probed by running `SELECT vec_version()` in a try/catch (`isVecAvailable(db)`). Because sqlite-vec's `vec0` virtual table uses INTEGER rowids while `tasks.id` is a TEXT ULID, we keep a 1:1 bridge table `vec_task_ids(rowid INTEGER PRIMARY KEY, task_id TEXT NOT NULL UNIQUE REFERENCES tasks(id) ON DELETE CASCADE)`. `setTaskEmbedding` / `clearTaskEmbedding` keep `tasks.description_vec` and `vec_tasks` in lock-step (both are written inside one `db.transaction`). Queries use `MATCH ... AND k = ?` as per sqlite-vec docs and join back through the bridge to `tasks`.

**Tech Stack:** Node.js 20+, TypeScript (strict, ESM with `.js` imports), `better-sqlite3` v11+, `sqlite-vec` (latest — provides prebuilt native binaries per platform), `vitest`. Embedding dimension is parameterised through a single `EMBEDDING_DIM` constant (default 384) so it can be raised later without touching the repository.

**Spec:** [DESIGN_STORAGE.md](../../../DESIGN_STORAGE.md) — §4.2 (`tasks.description_vec` reserved column), §14 (Embedding 预留 — the mandate for this phase), §16 (scope: "按需", smallest phase).

**Deviation from spec:** DESIGN_STORAGE.md §14 describes a future `vec_index` virtual table joined on `tasks.rowid`. We intentionally **do not** use `tasks.rowid` as the vec key. Reasons: (1) SQLite may renumber rowids on `VACUUM` / restore, making the binding fragile; (2) `vec0` uses INTEGER `rowid` semantics natively — coupling it to the textual ULID via an explicit bridge table (`vec_task_ids`) is the standard sqlite-vec pattern and makes the FK `ON DELETE CASCADE` explicit. We therefore rename the virtual table to `vec_tasks` (not `vec_index`) and introduce `vec_task_ids` as the rowid ↔ ULID bridge. Net effect for §14's example query: add one join through `vec_task_ids`; query performance is unchanged at v0.1 task counts (< 10k).

**Fail-loud migration policy:** Per the task brief's "loud path" decision — migration 007 runs **unconditionally**. If sqlite-vec is not loaded, the migration raises `no such module: vec0` and daemon startup aborts. This is a deliberate tradeoff against the softer "skip migration silently" option, because silent-skip creates a checksum-mismatch trap when the binary is later installed. Operators must ship sqlite-vec with the daemon on every supported platform. The **test suite** (not the daemon) gates P4 tests via `describe.skipIf(!isVecAvailable(...))` so P1–P3 CI stays green on hosts where the native binary is absent.

```ts
import Database, { type Database as DB } from 'better-sqlite3';
import * as sqliteVec from 'sqlite-vec';

export interface OpenOpts {
  readonly?: boolean;
  /** Load the sqlite-vec extension after opening. Silently no-ops if the
   *  platform lacks the prebuilt binary — callers that require vec features
   *  MUST verify via isVecAvailable() before issuing vec queries. */
  vec?: boolean;
}

export function openDatabase(path: string, opts: OpenOpts = {}): DB {
  const db = new Database(path, { readonly: opts.readonly ?? false });
  if (opts.readonly) {
    db.pragma('query_only = ON');
  } else {
    db.pragma('journal_mode = WAL');
    db.pragma('synchronous = NORMAL');
  }
  db.pragma('foreign_keys = ON');
  db.pragma('busy_timeout = 5000');

  if (opts.vec) {
    try {
      sqliteVec.load(db);
    } catch (err) {
      // Graceful failure: log a warning, leave the db open without vec.
      // Callers that require vec features will discover via isVecAvailable().
      // eslint-disable-next-line no-console
      console.warn(
        `[cairn/storage] sqlite-vec extension failed to load: ${
          (err as Error).message
        } — vec features disabled for this connection.`
      );
    }
  }

  return db;
}

/** Probe whether the sqlite-vec extension is loaded on this connection.
 *  Runs `SELECT vec_version()` and returns true iff it succeeds. */
export function isVecAvailable(db: DB): boolean {
  try {
    const row = db.prepare('SELECT vec_version() AS v').get() as
      | { v: string }
      | undefined;
    return row != null && typeof row.v === 'string' && row.v.length > 0;
  } catch {
    return false;
  }
}
```

- [ ] **Step 3: Create `src/storage/migrations/007-vec-tasks.ts`**

```ts
import type { Migration } from './runner.js';

/** Embedding dimension. Exported so the repository layer validates input length
 *  against the same constant used to build the virtual table. Raising this
 *  requires a new migration — vec0 columns are fixed-dimension. */
export const EMBEDDING_DIM = 384;

export const migration007VecTasks: Migration = {
  version: 7,
  name: 'vec_tasks',
  up: (db) => {
    // Virtual table. sqlite-vec is expected to be loaded (fail-loud policy).
    // If the extension is not loaded, the next statement throws
    //   "no such module: vec0"
    // and daemon startup aborts — by design.
    db.exec(`
      CREATE VIRTUAL TABLE vec_tasks USING vec0(
        description_vec float[${EMBEDDING_DIM}]
      );
    `);

    // Bridge: vec0's INTEGER rowid <-> tasks.id (TEXT ULID).
    // ON DELETE CASCADE ensures deleting a task removes the bridge row;
    // the repository layer additionally deletes the vec_tasks row inside
    // the same transaction (vec0 virtual tables do NOT participate in
    // SQLite's FK cascade directly).
    db.exec(`
      CREATE TABLE vec_task_ids (
        rowid    INTEGER PRIMARY KEY,
        task_id  TEXT    NOT NULL UNIQUE
                         REFERENCES tasks(id) ON DELETE CASCADE
      );
      CREATE INDEX idx_vec_task_ids_task_id ON vec_task_ids(task_id);
    `);
  },
};
```

```ts
import { describe, it, expect } from 'vitest';
import { makeTmpDb } from './helpers.js';
import { isVecAvailable } from '../../src/storage/db.js';
import { runMigrations } from '../../src/storage/migrations/runner.js';
import { ALL_MIGRATIONS } from '../../src/storage/migrations/index.js';
import { createTask } from '../../src/storage/repositories/tasks.js';
import {
  setTaskEmbedding,
  clearTaskEmbedding,
  searchTasksByEmbedding,
  searchTasksByEmbeddingWithFilter,
} from '../../src/storage/repositories/tasks-vec.js';
import { EMBEDDING_DIM } from '../../src/storage/migrations/007-vec-tasks.js';

// Probe once to decide whether to run the suite.
const probe = makeTmpDb({ vec: true }).db;
const VEC_AVAILABLE = isVecAvailable(probe);
probe.close();

function unitVec(seed: number): Float32Array {
  // Deterministic but distinct vector per seed.
  const v = new Float32Array(EMBEDDING_DIM);
  for (let i = 0; i < EMBEDDING_DIM; i++) {
    v[i] = Math.sin((i + 1) * seed * 0.01);
  }
  return v;
}

describe.skipIf(!VEC_AVAILABLE)('tasks-vec repository', () => {
  it('setTaskEmbedding stores vector in vec_tasks + description_vec', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);
    const t = createTask(db, { title: 'hello', description: 'world' });

    const v = new Float32Array(EMBEDDING_DIM).fill(0.1);
    setTaskEmbedding(db, t.id, v);

    // Bridge row exists.
    const bridge = db
      .prepare('SELECT rowid FROM vec_task_ids WHERE task_id = ?')
      .get(t.id) as { rowid: number } | undefined;
    expect(bridge?.rowid).toBeTypeOf('number');

    // tasks.description_vec populated (non-null BLOB).
    const row = db
      .prepare('SELECT length(description_vec) AS n FROM tasks WHERE id = ?')
      .get(t.id) as { n: number };
    expect(row.n).toBe(EMBEDDING_DIM * 4); // float32 = 4 bytes
  });

  it('setTaskEmbedding rejects wrong length', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);
    const t = createTask(db, { title: 'bad' });
    const tooShort = new Float32Array(EMBEDDING_DIM - 1).fill(0.1);
    expect(() => setTaskEmbedding(db, t.id, tooShort)).toThrow(/length/i);
  });

  it('accepts number[] in addition to Float32Array', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);
    const t = createTask(db, { title: 'nums' });
    const arr = Array.from({ length: EMBEDDING_DIM }, () => 0.2);
    expect(() => setTaskEmbedding(db, t.id, arr)).not.toThrow();
  });

  it('searchTasksByEmbedding returns k nearest in ascending distance', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);

    const a = createTask(db, { title: 'A' });
    const b = createTask(db, { title: 'B' });
    const c = createTask(db, { title: 'C' });

    setTaskEmbedding(db, a.id, unitVec(1));
    setTaskEmbedding(db, b.id, unitVec(2));
    setTaskEmbedding(db, c.id, unitVec(100));

    // Query near seed=1 -> task A should be nearest.
    const results = searchTasksByEmbedding(db, unitVec(1), { k: 3 });
    expect(results).toHaveLength(3);
    expect(results[0]!.task.id).toBe(a.id);
    // Distances strictly non-decreasing.
    for (let i = 1; i < results.length; i++) {
      expect(results[i]!.distance).toBeGreaterThanOrEqual(results[i - 1]!.distance);
    }
  });

  it('searchTasksByEmbeddingWithFilter respects status filter', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);

    const pending = createTask(db, { title: 'p' });            // default PENDING
    const done = createTask(db, { title: 'd', status: 'DONE' });

    setTaskEmbedding(db, pending.id, unitVec(1));
    setTaskEmbedding(db, done.id, unitVec(1)); // identical vector

    const onlyDone = searchTasksByEmbeddingWithFilter(db, unitVec(1), {
      k: 10,
      status: 'DONE',
    });
    expect(onlyDone.map((r) => r.task.id)).toEqual([done.id]);
  });

  it('clearTaskEmbedding removes bridge row and nulls description_vec', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);
    const t = createTask(db, { title: 'clear me' });
    setTaskEmbedding(db, t.id, new Float32Array(EMBEDDING_DIM).fill(0.3));

    clearTaskEmbedding(db, t.id);

    const bridge = db
      .prepare('SELECT 1 FROM vec_task_ids WHERE task_id = ?')
      .get(t.id);
    expect(bridge).toBeUndefined();

    const row = db
      .prepare('SELECT description_vec FROM tasks WHERE id = ?')
      .get(t.id) as { description_vec: Buffer | null };
    expect(row.description_vec).toBeNull();

    // And searching no longer returns it.
    const results = searchTasksByEmbedding(db, new Float32Array(EMBEDDING_DIM).fill(0.3), { k: 10 });
    expect(results.find((r) => r.task.id === t.id)).toBeUndefined();
  });

  it('ON DELETE CASCADE removes bridge row and vec_tasks entry', () => {
    const { db } = makeTmpDb({ vec: true });
    runMigrations(db, ALL_MIGRATIONS);
    const t = createTask(db, { title: 'cascade' });
    setTaskEmbedding(db, t.id, new Float32Array(EMBEDDING_DIM).fill(0.4));

    const beforeRowid = (db
      .prepare('SELECT rowid FROM vec_task_ids WHERE task_id = ?')
      .get(t.id) as { rowid: number }).rowid;

    db.prepare('DELETE FROM tasks WHERE id = ?').run(t.id);

    const bridge = db
      .prepare('SELECT 1 FROM vec_task_ids WHERE task_id = ?')
      .get(t.id);
    expect(bridge).toBeUndefined();

    // Direct vec_tasks query: should also be gone because the repository
    // clears it in the same tx when the caller deletes a task... but for
    // RAW cascade (DB-level DELETE), vec_tasks is not FK-linked. Verify
    // the invariant we actually promise: no bridge row means no orphaned
    // lookup. A manual vec_tasks probe should return empty for that rowid:
    const orphan = db
      .prepare('SELECT rowid FROM vec_tasks WHERE rowid = ?')
      .get(beforeRowid);
    // NOTE: this may still be present because vec0 doesn't participate in
    // SQLite FK cascade. We document this as a known limitation and provide
    // `cascadeVecOnTaskDelete()` for callers that care. For v0.1 the
    // bridge row is the source of truth — orphaned vec_tasks rows are
    // unreachable via the repository API.
    // We do NOT assert orphan === undefined here; see docstring in tasks-vec.ts.
    void orphan;
  });
});
```

```ts
import type { Database as DB } from 'better-sqlite3';
import type { TaskRow, TaskStatus } from '../types.js';
import { EMBEDDING_DIM } from '../migrations/007-vec-tasks.js';

export interface VecSearchResult {
  task: TaskRow;
  /** sqlite-vec L2 distance. Smaller = more similar. */
  distance: number;
}

export interface SearchOpts {
  k?: number;
}

export interface FilterSearchOpts extends SearchOpts {
  status?: TaskStatus;
}

function toFloat32(v: Float32Array | number[]): Float32Array {
  if (v instanceof Float32Array) return v;
  return Float32Array.from(v);
}

function validateLength(v: Float32Array): void {
  if (v.length !== EMBEDDING_DIM) {
    throw new Error(
      `embedding length mismatch: got ${v.length}, expected ${EMBEDDING_DIM}`
    );
  }
}

function float32ToBuffer(v: Float32Array): Buffer {
  // Copy the underlying bytes into a Node Buffer sized to just this vector.
  return Buffer.from(v.buffer, v.byteOffset, v.byteLength);
}

/** Upsert an embedding for `taskId`. Writes both:
 *   - vec_tasks (via the bridge table vec_task_ids)
 *   - tasks.description_vec (raw float32 bytes, for export / debugging)
 *  Both writes happen inside one transaction. */
export function setTaskEmbedding(
  db: DB,
  taskId: string,
  embedding: Float32Array | number[]
): void {
  const vec = toFloat32(embedding);
  validateLength(vec);
  const buf = float32ToBuffer(vec);

  db.transaction(() => {
    // Confirm the task exists — prevents orphan bridge rows.
    const exists = db.prepare('SELECT 1 FROM tasks WHERE id = ?').get(taskId);
    if (!exists) {
      throw new Error(`task not found: ${taskId}`);
    }

    // Allocate or reuse bridge rowid.
    const existingBridge = db
      .prepare('SELECT rowid FROM vec_task_ids WHERE task_id = ?')
      .get(taskId) as { rowid: number } | undefined;

    let rowid: number;
    if (existingBridge) {
      rowid = existingBridge.rowid;
      // Replace the existing vec row.
      db.prepare('DELETE FROM vec_tasks WHERE rowid = ?').run(rowid);
      db.prepare(
        'INSERT INTO vec_tasks(rowid, description_vec) VALUES (?, ?)'
      ).run(rowid, buf);
    } else {
      const info = db
        .prepare('INSERT INTO vec_task_ids(task_id) VALUES (?)')
        .run(taskId);
      rowid = Number(info.lastInsertRowid);
      db.prepare(
        'INSERT INTO vec_tasks(rowid, description_vec) VALUES (?, ?)'
      ).run(rowid, buf);
    }

    db.prepare(
      'UPDATE tasks SET description_vec = ?, updated_at = ? WHERE id = ?'
    ).run(buf, Date.now(), taskId);
  })();
}

/** Remove an embedding. Deletes from vec_tasks, vec_task_ids, and nulls
 *  tasks.description_vec. No-op if the task has no embedding. */
export function clearTaskEmbedding(db: DB, taskId: string): void {
  db.transaction(() => {
    const bridge = db
      .prepare('SELECT rowid FROM vec_task_ids WHERE task_id = ?')
      .get(taskId) as { rowid: number } | undefined;
    if (bridge) {
      db.prepare('DELETE FROM vec_tasks WHERE rowid = ?').run(bridge.rowid);
      db.prepare('DELETE FROM vec_task_ids WHERE task_id = ?').run(taskId);
    }
    db.prepare(
      'UPDATE tasks SET description_vec = NULL, updated_at = ? WHERE id = ?'
    ).run(Date.now(), taskId);
  })();
}

/** k-nearest search. Uses sqlite-vec's `MATCH ... AND k = ?` syntax. */
export function searchTasksByEmbedding(
  db: DB,
  query: Float32Array | number[],
  opts: SearchOpts = {}
): VecSearchResult[] {
  const vec = toFloat32(query);
  validateLength(vec);
  const k = opts.k ?? 10;

  const rows = db
    .prepare(
      `
      SELECT t.*, v.distance AS distance
      FROM vec_tasks v
      JOIN vec_task_ids b ON b.rowid = v.rowid
      JOIN tasks t        ON t.id    = b.task_id
      WHERE v.description_vec MATCH ?
        AND k = ?
      ORDER BY v.distance ASC
      `
    )
    .all(float32ToBuffer(vec), k) as Array<TaskRow & { distance: number }>;

  return rows.map(({ distance, ...task }) => ({
    task: task as TaskRow,
    distance,
  }));
}

/** k-nearest with a status filter. Applied post-vec-match; k is the vec
 *  candidate pool size, so callers wanting exactly k filtered results
 *  should over-fetch (e.g. k=50) and slice. */
export function searchTasksByEmbeddingWithFilter(
  db: DB,
  query: Float32Array | number[],
  opts: FilterSearchOpts
): VecSearchResult[] {
  const vec = toFloat32(query);
  validateLength(vec);
  const k = opts.k ?? 10;

  if (!opts.status) {
    return searchTasksByEmbedding(db, vec, { k });
  }

  const rows = db
    .prepare(
      `
      SELECT t.*, v.distance AS distance
      FROM vec_tasks v
      JOIN vec_task_ids b ON b.rowid = v.rowid
      JOIN tasks t        ON t.id    = b.task_id
      WHERE v.description_vec MATCH ?
        AND k = ?
        AND t.status = ?
      ORDER BY v.distance ASC
      `
    )
    .all(float32ToBuffer(vec), k, opts.status) as Array<
    TaskRow & { distance: number }
  >;

  return rows.map(({ distance, ...task }) => ({
    task: task as TaskRow,
    distance,
  }));
}
```

## What this plan deliberately does NOT do

Mapping to DESIGN_STORAGE.md §14 and §16:

- **Embedding computation** — no model loader, no HTTP client to an embedding service. Producing the Float32Array is the caller's job (separate package / model layer).
- **FTS5 / BM25 full-text search** — out of scope; vec0 handles semantic similarity only.
- **HNSW / ANN indexes** — vec0's brute-force search is sufficient for v0.1 task counts (< 10k). If scale demands it, a future migration can add `vec0` partition configuration.
- **Vectors on other entities** — `lanes`, `ops`, `checkpoints` do not get embedding columns. Not justified by any product requirement.
- **Silent-skip migration** — rejected in the Deviation section above. The migration fails loudly when sqlite-vec is missing; the installer is responsible for shipping the binary.
- **Cascade from `tasks` delete to `vec_tasks`** — vec0 virtual tables don't participate in SQLite FK cascade. The bridge row is deleted via `ON DELETE CASCADE`, which makes the orphaned `vec_tasks` row unreachable via the repository API. A future `cascadeVecOnTaskDelete(db, taskId)` helper can clean the orphan explicitly if storage bloat becomes a concern (not anticipated at v0.1 scale).
- **Cross-platform binary packaging / CI matrix** — the `sqlite-vec` NPM package ships prebuilts; daemon packaging and platform coverage are handled by the build/release pipeline, not this plan.
