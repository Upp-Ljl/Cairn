## 1. LLM 失败（7 条）

### 1.1 rate limit

**触发条件**：daemon 在短时间内向 LLM API 发送大量请求（如多个 sub-agent 并行执行，触发 API provider 的 RPM / TPM 限制）。

**当前行为**：API 返回 429 Too Many Requests，daemon 未处理时子 Agent 报错并将任务标为 FAILED。

**应有行为**：daemon 捕获 429 响应，自动退避重试（exponential backoff，初始 1s，最大 60s），任务状态切换为 PAUSED（等待重试），IPC 向 UI 推送 "rate_limit_backoff" 事件含预估等待时间；超过 5 次重试仍失败时才将任务标为 FAILED。

**验证方法**：
- 单元测试：mock LLM API 返回 429，验证 daemon 触发退避逻辑（日志包含 'rate limit backoff, attempt N'）
- 单元测试：5 次重试后仍 429，验证任务状态变为 FAILED，IPC 发送 "task_failed" 事件

**处理优先级**：v0.1 必修（多 sub-agent 并行场景下极易触发，不处理会导致整个任务树崩溃）

---

### 1.2 token 超限

**触发条件**：单次 LLM API 请求的 prompt 超过模型 context window 限制（如 Claude 3.5 Sonnet 的 200K token 上限），通常发生在 conversation 很长时（L2 memory checkpoint 未及时 truncate）或附带大量文件内容时。

**当前行为**：API 返回 400 Bad Request（error message 包含 "max tokens"），daemon 未处理时 sub-agent 报错。

**应有行为**：daemon 在发送前估算 token 数（simple 字符数 / 4 估算），若估算超过 model_context_limit * 0.9，自动触发 L2 conversation truncate（截断最老的 messages）并重试。若截断后仍超限，任务进入 PAUSED，IPC 向 UI 推送 "context_overflow" 事件，提示用户"本次对话已过长，建议重新开始或手动 rewind"。

**验证方法**：
- 单元测试：mock LLM API 返回 400 with "max tokens"，验证 daemon 触发 L2 truncate 并重试
- 单元测试：truncate 后仍超限，验证任务进入 PAUSED 并发送 context_overflow 事件

**处理优先级**：v0.1 必修（长任务场景必然触发，不处理用户体验崩溃）

---

### 1.3 cost cap（API 费用上限触发）

**触发条件**：用户在 API provider 设置了月度费用上限，某次请求触发后所有后续请求返回 403 / 429（具体 code 因 provider 而异）。

**当前行为**：视为普通 API 错误，daemon 可能重试多次无效。

**应有行为**：daemon 识别 cost cap 触发的特定错误码（Anthropic: `credit_balance_too_low`；OpenAI: `insufficient_quota`），立即停止所有 sub-agent 的 LLM 调用（不重试），将所有 RUNNING 任务标为 PAUSED，IPC 向 UI 推送 "api_quota_exhausted" 事件，提示用户"API 余额不足，请检查账户"。

**验证方法**：
- 单元测试：mock API 返回 `credit_balance_too_low`，验证 daemon 不重试，所有 RUNNING 任务变为 PAUSED，IPC 发送 api_quota_exhausted
- 单元测试：mock API 返回 `insufficient_quota`（OpenAI），同上

**处理优先级**：v0.1 必修（否则用户账户触发 cap 时 daemon 会反复重试，白白耗时）

---

### 1.4 hallucination（模型输出与事实不符）

**触发条件**：sub-agent 的 LLM 输出中包含不存在的文件路径、错误的函数签名、虚构的 API 等，导致后续工具调用失败。

**当前行为**：工具调用失败（如 `file not found`），sub-agent 可能自我修正或将错误传播给 Outward Agent。

**应有行为**：这是 LLM 的固有局限，v0.1 不在 daemon 层做 hallucination 检测。处理策略：工具调用失败时将错误信息原样返回给 sub-agent，让模型自我修正；连续 3 次相同类型错误（如连续 3 次 `file not found`）时任务进入 PAUSED，提示用户介入。

**验证方法**：
- 单元测试：mock 工具调用连续返回同类错误 3 次，验证任务进入 PAUSED

**处理优先级**：监控并接受（hallucination 是模型层面问题，v0.1 只做基础兜底，深度检测推 v0.2）

---

### 1.5 cancel-mid-stream（用户在流式输出中途取消）

**触发条件**：用户在 LLM 流式响应进行中点击"取消"或关闭 UI，daemon 需要清理半完成的流式处理。

**当前行为**：流式连接关闭，daemon 侧 sub-agent 进程可能继续等待已断开的流，内存泄漏。

**应有行为**：UI 发送 IPC cancel 请求 → daemon 立即中止对应 sub-agent 的 LLM stream（调用 `AbortController.abort()`）→ 已写入文件的内容由最近 checkpoint 保护（用户可 rewind）→ 任务状态切换为 PAUSED（不是 FAILED），提示"用户取消，可继续或 rewind"。

**验证方法**：
- 单元测试：向 daemon 发 IPC cancel 请求，验证 LLM stream 被 abort（不继续消费 stream）且 sub-agent 进程退出（无僵尸进程）

**处理优先级**：v0.1 必修（用户取消是高频操作，不处理会造成进程泄漏和状态不一致）

---

### 1.6 API 不可用（LLM provider 宕机 / 网络中断）

**触发条件**：用户网络断开或 LLM provider（如 Anthropic）服务宕机，API 请求超时或返回 5xx。

**当前行为**：请求超时，sub-agent 报错，任务可能标为 FAILED。

**应有行为**：daemon 将网络不可用的错误与 4xx 类错误区分：5xx / 网络超时触发退避重试（最多 3 次，间隔 5s / 10s / 20s）；重试后仍失败时任务进入 PAUSED，IPC 向 UI 推送 "api_unavailable" 事件，提示"网络或 API 服务不可用，daemon 将在网络恢复后继续（每 60s 检测一次）"。

**验证方法**：
- 单元测试：mock API 返回 503，验证退避重试逻辑（日志含 'api unavailable, retry N of 3'）
- 单元测试：3 次重试后仍失败，验证任务进入 PAUSED，IPC 发送 api_unavailable

**处理优先级**：v0.1 必修（网络中断是真实运行场景，不处理任务会大量进入 FAILED 状态）

---

### 1.7 模型行为漂移（不同版本 API 返回格式变化）

**触发条件**：LLM provider 更新模型（如从 claude-3-5-sonnet-20241022 切换到新版本），新版本的工具调用 JSON 格式、system prompt 响应格式或 stop reason 字段发生变化，导致 daemon 解析失败。

**当前行为**：daemon 解析 LLM 响应时 throw，sub-agent 崩溃，任务 FAILED。

**应有行为**：daemon 在解析 LLM 响应时使用防御性解析（optional chaining，parse error 时记录原始响应）；解析失败时将 raw response 写入日志（不含 API key），任务进入 PAUSED，提示用户"模型响应格式异常，可能是 API 版本变化，请查看 `~/.cairn/logs/` 中的原始响应"。

**验证方法**：
- 单元测试：mock LLM 返回格式缺少必要字段（如 `stop_reason` 缺失），验证 daemon 不崩溃，日志包含 raw response（脱敏后）

**处理优先级**：v0.2（模型版本锁定在 v0.1 配置中，短期不会轮换；v0.2 引入模型版本管理后一并处理）

## 2. 存储 / 并发失败（5 条）

### 2.1 SQLite 损坏

**触发条件**：SQLite 数据库文件（`~/.cairn/cairn.db`）损坏，通常由以下原因引起：意外断电（WAL 文件未 checkpoint）、磁盘 I/O 错误、文件系统损坏、用户直接编辑 DB 文件。

**当前行为**：daemon 启动时 `better-sqlite3` 抛出 `SQLITE_CORRUPT` 错误，daemon 崩溃并反复重启（watchdog 循环）。

**应有行为**：
1. daemon 启动时捕获 `SQLITE_CORRUPT`，立即停止 watchdog 循环（不反复重启）
2. 尝试 `PRAGMA integrity_check` 定位损坏范围
3. 若只有 WAL 文件损坏：执行 `PRAGMA wal_checkpoint(TRUNCATE)` 后重试
4. 若主库损坏：将 `cairn.db` 重命名为 `cairn.db.corrupted-{timestamp}`，从最近备份恢复（v0.1 先实现"没有备份时报错并退出"，backup 机制推 v0.2）
5. 写入紧急日志到 `~/.cairn/logs/corruption-{timestamp}.log`，包含 integrity_check 输出

**验证方法**：
- 单元测试：向 `cairn.db` 写入垃圾字节使其损坏，启动 daemon，验证：不进入重启循环，日志包含 'SQLITE_CORRUPT detected'，DB 文件被重命名，daemon 以 exit 1 退出（v0.1 无备份情况）

**处理优先级**：v0.1 必修（DB 损坏不处理会让 watchdog 进入无限重启循环，消耗 CPU/磁盘）

---

### 2.2 磁盘满

**触发条件**：`~/.cairn/snapshots/` 目录或 `~/.cairn/logs/` 目录所在磁盘空间不足，rsync snapshot 写入失败，或日志 rotate 写入失败。

**当前行为**：文件写入返回 `ENOSPC`，snapshot 两阶段提交的 Phase 2 失败，checkpoint 残留在 PENDING 状态（待 CORRUPTED scan 清理）。

**应有行为**：
1. daemon 在 snapshot 写入前检查可用磁盘空间（估算：当前工作树大小 * 1.5）
2. 若可用空间 < 估算值，取消本次 snapshot，任务进入 PAUSED，IPC 推送 "disk_full" 事件，提示用户"磁盘空间不足（可用 N MB，需要 M MB），请清理后重试"
3. 若在写入过程中遇到 ENOSPC：Phase 2 异常退出，CORRUPTED scan 清理残留，checkpoint 标为 CORRUPTED（而非 PENDING 永远存在）

**验证方法**：
- 单元测试：mock 磁盘空间检查返回"不足"，验证 checkpoint 不开始写入，IPC 发送 disk_full 事件
- 单元测试：mock Phase 2 中途 ENOSPC，重启后 CORRUPTED scan 清理，checkpoint 状态变为 CORRUPTED

**处理优先级**：v0.1 必修（用户磁盘满是真实场景，不处理会让 checkpoint 永远悬挂在 PENDING 状态）

---

### 2.3 文件锁竞争

**触发条件**：多个 sub-agent 并行改同一个文件（如 sub-agent A 和 B 同时写 `src/utils.ts`），导致 git-stash 或 rsync snapshot 捕获到竞争状态，或 rewind 时文件版本不一致。

**当前行为**：无文件级锁保护，两个 sub-agent 可能同时写同一文件，最后写入者覆盖前者，数据静默丢失。

**应有行为**：daemon 的 Outward Agent 在分配 sub-agent 任务时，检查任务之间的文件修改范围（通过 sub-agent 的任务描述估算）；若有重叠文件，串行化这些 sub-agent（不并行）；若无法在任务分配阶段判断，任务执行时使用文件级 advisory lock（`flock`/`LockFileSync`）；冲突时挂起并向 UI 推送 "file_conflict" 事件，让用户决定。

**验证方法**：
- 单元测试：两个 sub-agent 并发尝试写同一文件，验证其中一个挂起等待，另一个完成后前者继续（串行）

**处理优先级**：v0.1 必修（无锁保护会导致数据静默丢失，违反"可回滚"的核心承诺；v0.1 先实现"冲突时串行"最保守策略）

---

### 2.4 写入中断（进程被强制 kill 或断电）

**触发条件**：在 checkpoint 两阶段提交的 Phase 2（文件 I/O）进行中，daemon 被 `kill -9` 或系统断电，导致 snapshot 目录写入不完整，DB 中该 checkpoint 仍为 PENDING。

**当前行为**：U-CORRUPT 已设计了 CORRUPTED scan 处理此场景（daemon 启动时扫描 PENDING 超过 5 分钟的 checkpoint → 标 CORRUPTED + 清理目录）。

**应有行为（验证要点）**：
1. 重启后 CORRUPTED scan 在 daemon 启动期自动执行
2. 被中断的 checkpoint 标为 CORRUPTED，不出现在 `listCheckpoints('READY')` 中
3. 残留的 snapshot 目录被 `rm -rf`（路径校验：只删 `~/.cairn/snapshots/{ckpt_id}/`，不越界）

**验证方法**：
- 单元测试（U-CORRUPT 中已覆盖）：模拟 Phase 2 中途 crash，重启后验证 CORRUPTED 标记和目录清理
- 额外验证：路径校验——`snapshot_dir` 包含 `../..` 时，清理操作拒绝执行（返回 error 而非执行 rm）

**处理优先级**：v0.1 必修（U-CORRUPT unit 已覆盖，本条作为持续验证项目）

---

### 2.5 跨进程 race（CLI 和 daemon 同时写 DB）

**触发条件**：用户同时运行 `cairn rewind <id>`（CLI 直连 DB，方案 A）和 daemon 正在执行任务（daemon 也在写 DB），两个写者同时持有写锁。

**当前行为**：SQLite WAL 模式下 `busy_timeout=5000ms`，等待 5s 后其中一个 writer 返回 `SQLITE_BUSY`，daemon/CLI 可能报错。

**应有行为**：
1. CLI 使用 `BEGIN EXCLUSIVE` 事务前先检查 daemon 是否有活跃任务（读 `tasks` 表中 `status='RUNNING'` 的行）
2. 若有活跃任务，CLI 提示"daemon 有进行中任务，rewind 可能影响其状态，是否继续？"并等待用户确认
3. `SQLITE_BUSY` 发生时 CLI 以清晰错误信息退出（exit 1 + stderr "database is busy, please try again"），不静默失败
4. `busy_timeout=5000ms` 作为最后兜底，在 5s 内等待，超时后明确报错

**验证方法**：
- 单元测试：模拟 CLI 和 daemon 同时写 DB（两个 `better-sqlite3` 连接同时 `BEGIN EXCLUSIVE`），验证其中一个在 5s 内 SQLITE_BUSY 并有明确错误信息
- 单元测试：CLI rewind 前有活跃任务时，CLI 输出确认提示

**处理优先级**：监控并接受（DESIGN_STORAGE.md §6.3 明确接受 lock_holder 短时冲突，由 CAS 状态机保护；v0.1 先接受偶发 SQLITE_BUSY，深度协调推 v0.2）

### 2.6 kill-9 race（watchdog 重启后旧实例 SQLite 文件锁未释放）

> v3 新增，reviewer Q3 触发

**触发条件**：daemon 进程被 `kill -9`（或 Windows `TerminateProcess`）强制终止后，watchdog 进程（外部 supervisor 或内置 watchdog）立即启动新实例；但旧实例的 SQLite 文件锁（`cairo.db-wal`、`cairn.db-shm` 或 Windows 文件锁）未能被 OS 及时释放，导致新 daemon 在短暂时间窗口（10–200ms）内无法获得 write lock，写入操作报 `SQLITE_BUSY` 或 `SQLITE_LOCKED`。

**当前行为**：新 daemon 启动时若 `better-sqlite3` 打开 DB 报 `SQLITE_BUSY`，daemon 以启动失败退出，watchdog 继续重试，陷入重启循环。

**应有行为**：
1. 新 daemon 启动时打开 DB 设 `busy_timeout = 3000ms`（3 秒），在此窗口内等待旧实例文件锁释放（通常 OS 在进程终止后 < 500ms 释放）
2. 3 秒内未获锁：daemon 写入紧急日志 `'db lock wait timeout after kill-9, retrying in 5s'`，等待 5 秒后再试一次
3. 两次均失败：daemon 以 exit 2 退出，watchdog 进入退避模式（`Restart=on-failure` with `RestartSec=10s`），日志包含 `'db unavailable after kill-9, entering backoff'`
4. 无论哪步，PID 文件在新实例成功写锁后才覆盖（避免新 PID 文件写入但 DB 未就绪）

**验证方法**：
- 单元测试：模拟 `kill -9`（进程 fork + SIGKILL），watchdog 重启新实例，新实例 3 秒内成功打开 DB（模拟旧锁在 1 秒后释放）
- 单元测试：模拟旧锁始终未释放（mock `busy_timeout` 不产生重试），验证新实例以 exit 2 退出，日志包含 `db unavailable after kill-9`

**处理优先级**：v0.1 必修（kill-9 + watchdog 是保证 daemon 高可用的核心机制，此 race 不处理会让 watchdog 陷入快速循环）

**关联 unit**：U-DAEMON-WATCHDOG（重启逻辑），U-DAEMON-PROC（PID 文件写入时机）

### 2.7 PAUSED 任务无限挂起（内存膨胀）

> v3 新增，reviewer Q3 触发

**触发条件**：子 Agent 任务进入 `PAUSED` 状态（等待用户决策）后，用户长时间未响应（如数小时或数天），任务相关的进程内存（进程缓冲、待发 IPC 队列、LLM stream 缓冲）持续占用，导致 daemon 进程内存随 PAUSED 任务积累而膨胀，最终 OOM 或严重影响系统性能。

**当前行为**：PAUSED 状态没有 timeout，任务可以无限等待。若 LLM stream 缓冲未释放，每个 PAUSED 任务持续占用 LLM 响应缓冲内存。

**应有行为**：
1. 子 Agent 进入 PAUSED 状态时，立即释放与该任务相关的 LLM stream 缓冲和临时数据
2. 任务进入 PAUSED 时记录 `paused_at` 时间戳（`sub_agents.paused_at`）
3. daemon 配置参数 `max_pause_timeout`（默认 30 分钟），超过此时间的 PAUSED 任务自动触发清理流程（U-DAEMON-SUBAGENT-KILL）并将状态标为 FAILED，IPC 向 UI 推送 `"task_pause_expired"` 事件
4. `max_pause_timeout = 0` 表示禁用（永不自动清理），仅允许在配置中显式设置

**验证方法**：
- 单元测试：任务进入 PAUSED 后，验证 LLM stream 缓冲已被释放（通过 mock 检查 AbortController 已 abort）
- 单元测试：mock `max_pause_timeout = 5s`，任务进入 PAUSED → 等待 6s → 验证任务状态变为 FAILED，`"task_pause_expired"` IPC 事件已发送

**处理优先级**：v0.1 必修（长时间运行的 daemon 场景下，PAUSED 任务积累必然导致内存问题；默认 30 分钟 timeout 覆盖日常使用场景）

**关联 unit**：U-DAEMON-SUBAGENT-SUPERVISE（PAUSED 状态检测），U-DAEMON-SUBAGENT-KILL（清理流程），U-DAEMON-CONFIG（`max_pause_timeout` 可配置）

## 3. 用户行为失败（4 条）

### 3.1 cancel-mid-retry（在 daemon 重试期间用户取消）

**触发条件**：daemon 在对 rate limit / 网络错误执行退避重试时，用户从 UI 发送 IPC cancel 请求，要求取消正在等待重试的任务。

**当前行为**：若 cancel 发生在 sleep 等待期间，取消可能不生效（sleep 不可中断）；任务继续重试直到成功或达到最大重试次数。

**应有行为**：
1. daemon 的退避重试逻辑使用可取消的 sleep（`Promise.race([sleep, cancelToken])`）
2. cancel 信号到达时，立即中断 sleep，不再发起重试
3. 任务状态切换为 PAUSED（用户显式取消），IPC 发送 "task_cancelled" 事件

**验证方法**：
- 单元测试：daemon 处于退避 sleep 中，发送 cancel → 验证 sleep 被中断（cancel 在 100ms 内生效），任务状态变为 PAUSED

**处理优先级**：v0.1 必修（用户取消是高频操作；退避等待最长 60s，不可中断会严重影响体验）

---

### 3.2 强制退出（用户强制 kill daemon 或系统强制关机）

**触发条件**：用户通过任务管理器强制结束 daemon 进程（`kill -9`），或系统强制关机（不触发 SIGTERM，直接断电）。

**当前行为**：daemon 立即死亡，PID 文件残留，`daemon.pid` 中的 PID 对应已死亡进程；下次 `cairn daemon start` 读到 PID 文件并认为 daemon 已在运行，拒绝启动。

**应有行为**：
1. `cairn daemon start` 读到 `daemon.pid` 后，执行 `kill -0 <pid>` 检测进程是否存活
2. 若进程不存在（kill -0 返回非 0），自动清理 `daemon.pid` 和 IPC socket 文件，并正常启动新实例
3. 日志记录 "stale PID file detected (PID {n} not running), cleaning up and starting fresh"
4. 强制退出后的进行中任务，在下次 daemon 启动时由 U-DAEMON-PERSIST 的恢复逻辑读回（status 仍为 RUNNING 或 PAUSED，可续跑）

**验证方法**：
- 单元测试：留下 stale PID 文件（对应一个不存在的 PID），执行 `cairn daemon start`，验证 stale PID 被清理，daemon 正常启动

**处理优先级**：v0.1 必修（stale PID 文件不清理会让用户无法重启 daemon，严重影响可用性）

---

### 3.3 同时操作多个 cairn 实例（多仓库 / 多 `~/.cairn` 并行）

**触发条件**：用户在两个不同仓库的终端同时运行 cairn 相关操作（如两个 MCP 楔同时活跃），或者用户手动设置了多个 `cairn` 配置目录（通过 `CAIRN_HOME` 环境变量）。

**当前行为**：两个 cairn 实例共享 `~/.cairn/cairn.db`，单实例锁（PID 文件）会阻止第二个 daemon 启动，但两个 MCP 楔（不是 daemon）可能同时写 SQLite。

**应有行为**：
1. 单实例 daemon 锁是设计决策（v0.1 只支持一个 daemon 实例），不改变
2. MCP 楔（非 daemon 模式）写 SQLite 时使用 WAL + busy_timeout，两个楔并发写时通过 SQLite 串行化
3. 日志中清晰标注是哪个仓库（`cwd` 路径）的操作，方便 debug 多仓库并发问题
4. 文档明示 v0.1 只支持一个 daemon 实例（多仓库场景需轮换使用 daemon，或等待 v0.2 多 daemon 支持）

**验证方法**：
- 单元测试：两个 MCP 楔并发写 scratchpad，验证 SQLite WAL 串行化正确（两个写入都成功，不互相覆盖）

**处理优先级**：监控并接受（v0.1 文档明示单 daemon 限制；多实例深度支持推 v0.2）

---

### 3.4 误删 .cairn 目录

**触发条件**：用户不慎执行 `rm -rf ~/.cairn`，删除整个 cairn 数据目录（包含 `cairn.db`、所有 snapshot 目录、日志文件）。

**当前行为**：daemon 下次启动时 `better-sqlite3` 重新创建空的 `cairn.db`，migration runner 重新初始化 schema；所有历史 checkpoint 和 scratchpad 数据永久丢失，无法恢复。

**应有行为**：
1. v0.1 不做自动备份（DESIGN_STORAGE.md §16 明确 backup 机制推 v0.2）
2. 文档明示"误删 `~/.cairn` 目录后历史数据不可恢复"，建议用户定期备份（`cp -r ~/.cairn ~/.cairn.bak`）
3. daemon 启动时检测 `~/.cairn` 缺失，创建新目录并写入首次运行提示日志（"No existing data found, starting fresh"）
4. 若用户有 git 仓库的 checkpoint，文件系统层面的 rewind 数据已丢失（snapshot 在 `~/.cairn/snapshots/`），但 git HEAD 本身未变（git 历史可手动恢复）

**验证方法**：
- 单元测试：删除测试 `~/.cairn` 目录后启动 daemon，验证：daemon 正常启动，日志包含 "starting fresh"，不崩溃

**处理优先级**：v0.1 必修（基础行为验证：误删后不崩溃，日志明示；深度备份推 v0.2）

## 4. 平台差异失败（4 条）

### 4.1 Windows 路径长度限制

**触发条件**：Windows 默认路径长度上限为 260 字符（MAX_PATH）；cairn snapshot 目录路径可能较长（`~/.cairn/snapshots/{26 char ULID}/`），若用户的 `~` 路径本身很长（如 `C:\Users\very-long-username\`），加上仓库内文件路径可能超过 260 字符。

**当前行为**：超长路径的文件操作返回 `ENAMETOOLONG` 或 Windows 错误码 `ERROR_PATH_TOO_LONG`，rsync 或 Node.js `fs` 调用失败。

**应有行为**：
1. daemon 启动时检测 Windows 系统是否已启用长路径支持（注册表 `Computer\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\FileSystem\LongPathsEnabled`）
2. 若长路径未启用，在日志中记录警告："Windows long path support not enabled. If snapshots fail, enable via gpedit.msc > Computer Configuration > Administrative Templates > System > Filesystem > Enable Win32 long paths"
3. snapshot 路径超过 240 字符时（保留 20 字符余量），提前报错（exit snapshot with error 'snapshot path too long'），不静默失败

**验证方法**：
- 单元测试（Windows only）：模拟超长路径（240+ 字符），验证 daemon 给出清晰错误而非 crash
- 手动验证：在实际 Windows 开发机上确认 ULID-based snapshot 路径不超限

**处理优先级**：v0.1 必修（Windows 是主开发平台，路径超限不处理会有静默 I/O 失败）

---

### 4.2 macOS Gatekeeper 隔离

**触发条件**：用户从 GitHub Releases 下载 Cairn Tauri 应用（非通过 App Store / notarized 渠道），macOS Gatekeeper 标记应用为"来自不受信任的开发者"，阻止启动（quarantine flag）。

**当前行为**：应用无法启动，显示"'Cairn' cannot be opened because it is from an unidentified developer"。

**应有行为**：
1. v0.1 发布物包含安装说明：
   - 选项 A：用户右键 → 打开（绕过 Gatekeeper 一次）
   - 选项 B：`xattr -d com.apple.quarantine /Applications/Cairn.app`
   - 选项 C：长期方案是申请 Apple Developer ID 并 notarize（推 v0.2 正式发布时处理）
2. README 明确标注 macOS 安装注意事项（v0.1 工程交付闸第 10 条）

**验证方法**：
- 手动验证（macOS）：下载 .dmg 后按 README 步骤安装，验证应用可启动
- 确认 README 包含 Gatekeeper 绕过说明

**处理优先级**：v0.2（v0.1 是工程交付阶段，无正式 notarized 发布物；README 说明作为 v0.1 临时措施）

### 4.3 Linux 信号差异

**触发条件**：Linux 上 daemon 收到的信号行为与 macOS 有细微差异（如 `SIGHUP` 的默认行为、`SIGPIPE` 的处理）；Node.js 进程信号处理在不同 Linux 发行版上可能有差异。

**当前行为**：daemon 的信号处理代码（SIGTERM / SIGINT / SIGHUP）在 Windows + macOS 上验证，Linux 未单独验证。

**应有行为**：
1. daemon 在 Linux 上显式注册 SIGPIPE 处理（不使用默认的进程终止行为，而是记录错误后继续运行）
2. SIGHUP 在 Linux 上触发配置重载（U-DAEMON-CONFIG 中已定义），行为与 macOS 一致
3. 通过 U-CROSS-PLAT 的 Ubuntu LTS 平台冒烟验证基本信号处理正确

**验证方法**：
- U-CROSS-PLAT 中的 Ubuntu LTS 冒烟：验证 daemon start/stop/restart 在 Linux 上正常工作
- 单元测试（Linux / Ubuntu）：发 SIGHUP → 验证配置重载而非进程退出

**处理优先级**：v0.1 必修（Linux 是 U-CI 的 CI 运行平台，信号处理不对会导致 CI 测试行为异常）

---

### 4.4 文件名大小写敏感

**触发条件**：macOS / Windows 默认文件系统不区分大小写（HFS+ / NTFS），Linux 默认区分大小写（ext4）。用户在 macOS 上创建了同名但大小写不同的文件（如 `Utils.ts` 和 `utils.ts`），在 Linux 上这两个文件是不同文件，但在 macOS 上视为同一文件；snapshot 路径和 rewind 时可能出现文件覆盖。

**当前行为**：rsync snapshot 在 macOS 上可能只保存其中一个文件（大小写不敏感 → 路径冲突），rewind 到 Linux 上时另一个文件不存在。

**应有行为**：
1. v0.1 不做跨平台大小写冲突检测（技术复杂，v0.2 处理）
2. snapshot 时记录文件的原始路径（区分大小写），rewind 时按原始路径还原
3. 若检测到同一 snapshot 目录中有大小写冲突的路径，在日志中记录警告，不崩溃

**验证方法**：
- 单元测试（macOS / Linux 各一）：snapshot 包含大小写差异路径，验证在各自平台上 restore 不崩溃

**处理优先级**：监控并接受（大小写冲突是边缘场景，用户需要主动制造；v0.1 记录警告，不深度处理）

## 5. 时钟失败（2 条）

### 5.1 系统时间倒拨

**触发条件**：用户手动修改系统时钟（或 NTP 同步导致时钟突然回拨），导致 daemon 使用 `Date.now()` 的时间戳小于已存储在 SQLite 中的时间戳（如 `checkpoints.created_at`）。

**当前行为**：新创建的 checkpoint 的 `created_at` 时间戳早于已有 checkpoint，`checkpoint.list` 按 `created_at DESC` 排序时顺序错乱；CORRUPTED scan 的 5 分钟阈值（`created_at < now()-5min`）可能误扫描到正常 PENDING checkpoint（若时钟回拨超过 5 分钟）。

**应有行为**：
1. daemon 使用单调时钟（`process.hrtime.bigint()`）计算相对时间差，只用 `Date.now()` 记录人类可读时间戳
2. CORRUPTED scan 的阈值改为基于进程启动后的相对时间（不是 wall clock absolute 时间），避免时钟回拨误扫描
3. checkpoint list 排序改为 ULID 字典序（ULID 含时间信息，单调递增，不受时钟回拨影响）

**验证方法**：
- 单元测试：写入几条 checkpoint 后模拟时钟回拨（修改 `Date.now` mock），验证 CORRUPTED scan 不误标正常 PENDING checkpoint

**处理优先级**：v0.1 必修（CORRUPTED scan 的 5 分钟阈值用绝对 wall clock 判断，时钟回拨 > 5 分钟时会把正常 checkpoint 误标为 CORRUPTED，造成数据丢失）

---

### 5.2 时区切换

**触发条件**：用户在 daemon 运行期间切换时区（如从 UTC+8 切到 UTC-5），或在不同时区的机器间移动 `~/.cairn` 目录（如通过网盘同步，不推荐）。

**当前行为**：SQLite 中的时间戳均为 unix ms（UTC），不受时区影响；但日志文件名（`daemon-{date}.log`）按本地时区日期命名，时区切换可能导致日志文件名跳变。

**应有行为**：
1. 所有 SQLite 时间戳字段使用 unix ms（已是当前设计，不受时区影响，无需改变）
2. 日志文件名从本地时区日期改为 UTC 日期（`daemon-{UTC-date}.log`），避免时区切换导致日志文件重复或跳变
3. UI 展示时间时，从 unix ms 转换为用户本地时区（前端 `new Date(ts).toLocaleString()`）

**验证方法**：
- 单元测试：daemon 在模拟时区切换前后各写一条日志，验证两条日志均在正确的 UTC 日期文件中

**处理优先级**：v0.2（时区切换是罕见场景；unix ms 时间戳已是 UTC，核心数据不受影响；日志文件名改 UTC 是小优化，不阻塞 v0.1 工程交付）

---

### 5.3 rewind 与 LLM stream race（conversation 状态不一致）

> v3 新增，reviewer Q3 触发

**触发条件**：用户在 LLM 流式输出正在进行时（sub_agent 正在 yield tokens 并追加 `messages` 表）发起 `rewind.to` 操作，两个并发写操作：stream 写入（`appendMessage`）和 rewind 截断（`truncateConversationTo`）存在 race condition，导致 conversation 状态不一致（如截断后部分 stream token 仍被写入，或截断操作在 stream 写入间隙执行导致 seq 编号错乱）。

**当前行为**：`rewind.to` 和 `appendMessage` 各自使用 SQLite 事务，但两者之间无协调层，并发时 DB 行为取决于 SQLite WAL 模式的 FIFO 串行化——截断和 stream 写入在 DB 层面可能交错，stream 截断后的 message 可能被追加，seq 编号可能错乱。

**应有行为**：
1. daemon 维护一个 `conversation_lock` 信号量（每个 conversation 一个），stream 写入和 rewind 截断均须先获取此锁
2. 用户发起 `rewind.to` 时，daemon 首先向对应 sub_agent 发送 `PAUSE_STREAM` 信号，等待 sub_agent 确认已暂停当前 stream yield（最多等待 2s）
3. stream 暂停后，执行 `truncateConversationTo`（原子事务），完成后释放锁并向 sub_agent 发 `ABORT_STREAM`（不再继续 stream）
4. 若 sub_agent 在 2s 内未确认暂停，daemon 强制 abort stream（`AbortController.abort()`），再执行 truncate，并在日志中记录 `'forced stream abort before rewind'`

**验证方法**：
- 单元测试：并发执行 `appendMessage`（10ms 间隔循环写）和 `truncateConversationTo`（在 loop 中途发起），验证截断后 `messages` 表中无截断序号之后的行
- 单元测试：`rewind.to` 期间 mock sub_agent 在 1s 后确认 `PAUSE_STREAM`，验证 truncate 在确认后才执行

**处理优先级**：v0.1 必修（rewind 是核心功能，conversation 状态不一致会让"回到过去"的承诺失效；race condition 在 LLM stream 场景下并非低概率）

**关联 unit**：U-L2-MEM（truncate 实现），U-DAEMON-SUBAGENT-IPC-BRIDGE（stream 暂停信号），U-CONV-PERSIST（conversation 锁机制）
