### 阶段四：第一波 PoC（PoC-1 + PoC-2，commit `555dde0`）

#### PoC-1：race window 压测

脚本：`packages/daemon/scripts/poc-1-race-stress.mjs`（worker_threads 多 connection 并发 SQLite 写）

4 场景结果（各 1000 ops）：

| 场景 | N | success | SQLITE_BUSY | p99 ms | 综合 |
|---|---|---|---|---|---|
| baseline | 2 | 100% | 0 | 1.38 | PASS |
| modest | 5 | 100% | 0 | 3.92 | PASS |
| elevated | 10 | 100% | 0 | 5.91 | PASS |
| stress | 50 | 100% | 0 | 448.59 | FAIL on p99 |

细化 verdict：v0.1 ICP 并发上限 N≤10 全部 PASS；N=50 的 449ms 是 SQLite 单写者锁的架构天花板，记录为 v0.3 议题（跨机 / 共享 daemon / agent 市场场景需重新评估 storage backend）。

落 `docs/superpowers/plans/2026-04-29-poc-1-results.md`，回填 ARCHITECTURE.md 4 处 PoC-1 锚点。

**决策 27**：SQLite WAL + busy_timeout=5000ms 在 v0.1 ICP 内足够；conflict-detection 应用层可直接叠加。

#### PoC-2：git pre-commit hook 延迟

脚本：`packages/daemon/scripts/poc-2-conflict-check.mjs`（hook 本体）+ `poc-2-hook-bench.mjs`（bench harness）

5 场景结果：

| 场景 | staged paths | exit≠0 | p99 ms | 阈值 ms | 综合 |
|---|---|---|---|---|---|
| clean | 0 | 0 | 81.4 | 200 | PASS |
| small | 10 | 0 | 86.0 | 200 | PASS |
| medium | 100 | 0 | 85.3 | 200 | PASS |
| large | 1000 | 0 | 122.2 | 1000 | PASS |
| fail-open | DB 缺失 | 0 | 88.3 | — | PASS |

关键发现：Node 冷启动 ~70ms 是延迟主导；SQLite 边际成本 ~45μs/query（prepared statement 复用）；fail-open 路径 24ms 内 exit 0，架构承诺成立。

落 `docs/superpowers/plans/2026-04-29-poc-2-results.md`，回填 ARCHITECTURE.md 2 处 PoC-2 锚点。

**决策 28**：commit-after hook 路径在 v0.1 完全可用；Native 二进制（消除 Node 冷启动 ~70ms）记为 v0.2/v0.3 候选。

## 10. 反思

**转向速度 vs. 文档完整性**：今天一天完成了 v1 → v2 整个定位转向，13 个新决策，9 份文档新建或全量重写。这个速度是 subagent rotation 加持的——单个 sonnet 不可能在一天内完成这个量。但代价是每个文档都"刚写完"，还没有经过用户实际使用中的摩擦检验。

**4-persona stress-test 有效**：这是今天最有价值的方法论产出。12 条 gap 中 5 条 P0 当天落地，Jess 的 Stripe v3/v2 案例和张涛的 `shared/types.ts` 案例让 §5.2 / §5.4 从抽象变具体。后续作为制度坚持。

**"按语境校正"的成本**：用户一句"我用 MiniMax 的 coding plan"触发了 ADR-4 整体重写。这说明产品定位的假设（用谁的 LLM）和技术设计之间的传导路径要短——产品假设变了，技术决策应该当天跟上，不要等到实施时才发现。今天做到了。

**PoC 比预算快 10 倍**：PoC-1 + PoC-2 预算 1.5 天，实际 1.5 小时。这说明 PoC 设计本身的清晰度够（pre-impl-validation 文档的价值在这里体现），以及 daemon 已有 API 直接可用（不需要搭环境）。PoC-3 / PoC-4 如果同样顺利，W3 时间盒有很大富余，可以把 W4 编码 plan 写得更细。

**force push 是今天唯一的破坏性操作**：commit squash 后 force push 是不得不用的操作。已建 backup branch 保护，但这个操作本身提示：下次应在 day 结束前先 squash 再 push，而不是在已 push 历史上 squash。

**EOD PoC-3 partial：subagent rotation 验证度更高的一例**：runner（haiku 候选 → 实际 sonnet）+ scorer（sonnet）+ report-writer（sonnet）= 3 只独立 subagent 接力，主 agent 只做 verdict 决策 + ARCHITECTURE.md / 归档回填。整个 pipeline 端到端 ~45 分钟（runner 5 min + scorer 4 min + report-writer 3 min + 主 agent ~30 min 编辑）。比 4-persona stress-test 更纯粹的 subagent 流水线——每只 subagent 任务边界清晰且互不干扰，scorer 独立打分避免 self-bias。**这个模式可以模板化**：未来类似"批量评估"任务（比如 PoC-3 第二 provider 补跑、4-persona 复测、cross-provider 横评）都可以套这个 pipeline。

**降级方案的诚实性**：DeepSeek 没 key 时，没有硬塞替代方案（拒绝替 user 注册免费 provider），而是落 partial verdict + 明确标注"接口可移植性命题待补"。这比强行凑双 provider 数据更诚实，也避免了把未验证的命题写死进 ADR-4。partial 状态可控、可补，比"完整但有水分"的报告更好。
