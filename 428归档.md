## 8. 反思

**subagent rotation 有效但成本可见**：本次三轮 validation plan review 每轮都找出上一轮没发现的问题，fresh context 是真实优势。但三轮迭代 = 三次完整文档重写（767 → 914 → 1058 行），小项目来说属于过度工程——核心产品范围都还没稳定，就在做如此精细的验证计划。

**"范围先行"没被执行**：4-28 花了大量时间写验证计划，4-28 EOD 用户才敲定"cairn = 工具集"，4-29 早上又反转加回 agent。如果"v0.1 = 什么"在 W2 Day 1 就锁死，验证计划的三轮 review 就不会产生"架构层缺口"——因为那些缺口（Outward Agent 没有 validation unit）来自范围自身的未定。教训：**在写大量 spec 文档之前，先把最顶层的范围决策敲死，哪怕一句话"v0.1 是不是 Agent 系统"也要先答出来**。

**dogfood 路径有效**：scratchpad.delete 是 4-29 早间 dogfood 自然发现的功能缺口，而不是从需求文档里挖出来的。这种"用自己的工具做自己的开发"的模式，比静态需求分析更能找到真实痛点。

**TDD 执行零回归**：三个产品功能（clean-tree / paths / auto-checkpoint）和 scratchpad.delete，每次都是先写 failing test 再修，测试从 91 升到 132，无一退化。

**4-29 反转的代价**：一次反转 = PRODUCT.md / validation plan / failure-modes 三份文档需要第三次重写，加上 agent design doc 从零起草。在产品战略清晰之前，大量 spec 文档等于"可以丢弃的临时产物"。下次应该把文档写作延迟到范围真正锁定后。
