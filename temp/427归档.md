## 8. 反思

**做对的地方**：subagent rotation 依然有效——T2.4 JSDoc、三个 fix commit、边界 acceptance 各用 fresh subagent 保证了聚焦度，没有因上下文积累引起的偏移。TDD 严格执行（先写 failing test 再 fix）让 `3fa561b` 和 `2455770` 都在修复的同时零回归。Friction 即时记录的纪律——每次 dogfood 结束立刻写进文件，而不是"回头再整理"——让 10 条 friction 的分类在 T2.7 只花了很短时间。

EOD 后阶段的 subagent rotation 在战略层面同样有效：GitButler 战略评估 subagent 没有顺着用户的 3 条启示讲，而是用项目证据反驳 2/3，并额外挖出 3 条用户未提到的启示；策略评估 subagent 对路径 B/C 直接打 3/10 和 2/10 的低分而不和稀泥。W3 计划经过 4 轮迭代（起草 → review → 大改 → eng-review → 小改）质量明显高于第一版草稿。

**可以更好的地方**：push 阶段两次遭遇 TLS 抖动（`read: connection reset` / `SSL error`），重试才过；下次应该在确认 push 成功后再关会话，不要假设第一次 push 一定成功。`docs/rewind-demo.md` 是 T2.5 dogfood 时故意搞坏的临时文件，dogfood 结束后需要人工 `git rm` 清理，这一步没有被自动化也没有在计划里预留——以后 dogfood 临时文件应该在任务单里加"清理步骤"。

W3 计划 4 轮迭代的修改过程里，前两轮（自己 review + 大改）其实可以合并：直接第一步就调 `plan-eng-review` subagent，跳过"自己先 review 再改"的中间步骤，省一轮。W3 计划留 untracked 到周日整理是合理决定，但超过 1 周不 commit 的草稿在周末时可能难以回忆修改原因，建议在文件顶部加"未 commit 原因 + 留待时间"注释。

**两条非显而易见的 lesson**：

1. 楔工具不下传 subagent 这件事起初看起来是限制（subagent-driven dogfood 用不到楔），但 T2.4 的实验揭示它是产品定位的准确揭示：楔是"父 agent 对自身工作状态的本地备份"，和"把工作分发给 subagent"是两个不同层次的关注。subagent 做代码改动，父 agent 做回滚决策——两者的工具集本应分离。这次 dogfood 让我们把这个设计意图从隐性变成了 README 里的显性文字。

2. 用户指令"产品功能优先 / 推迟其他"是清晰的硬约束，一旦 subagent 拿到这条指令，会自动给出非常 opinionated 的判断（路径 A=9/10，B=3/10，C=2/10），比让 subagent 自己权衡所有因素更高效、更可信。战略层面的 hard prior 就该明示，不要让 subagent 猜用户的隐性偏好。
