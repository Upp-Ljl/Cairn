### 阶段二：Phase 1 — auto SESSION_AGENT_ID + auto paths（无单独 commit，代码并入 c22da3a）

dispatch 一只 sonnet 处理 Phase 1。目标是让 Claude Code 用户不用手动传 `agent_id`——MCP server 启动时自动算出当前会话的稳定 id，写进环境变量，工具调用时 fallback 取环境变量。

核心变动集中在 `packages/mcp-server/src/workspace.ts`，新增 `computeAgentId(cwd: string): string`：

1. 先用 `execSync('git rev-parse --show-toplevel', { timeout: 1000, stdio: ['ignore','pipe','ignore'] })` 拿 git 仓库根目录。超时或不在 git 仓库里则 catch，fallback 到 raw `cwd`。
2. 用 `os.hostname() + ':' + canonicalPath` 拼字符串，做 sha1，取前 12 位，前缀 `cairn-`。
3. 结果写入 `process.env['CAIRN_SESSION_AGENT_ID']`，整个进程生命周期共享。

用 git toplevel 而非 raw cwd 的理由：同一仓库 `packages/daemon/` 和 `packages/mcp-server/` 子目录里启动的 Claude 实例，应当被视为同一工作区里的两个 agent，共享 agentId 才能正确检测文件重叠冲突。raw cwd 会让它们算出不同 id，冲突检测失效。

MCP schema 同步放宽：`cairn.checkpoint.create`、`cairn.process.register` 等工具的 `agent_id` 字段从 `required` 降为 `optional with default`。后端逻辑：调用方传了就用传入值，没传就取 `process.env['CAIRN_SESSION_AGENT_ID']`，再没有就生成一个临时 ULID（保持向后兼容）。

sonnet 跑完汇报：daemon 207 / mcp-server 147 pass。Phase 1 自测断言只验证了 ULID 格式（后来 opus 指出这是 HIGH 问题，Phase 4 后修复）。

### 阶段三：Phase 3 — dispatch FORCE_FAIL + R6 + pre-commit 升级（daemon 部分为 commit 019c2d2）

一只 sonnet 承包 Phase 3 的大单子，改动横跨 daemon 存储层和 mcp-server 工具层，是本日改动最大的 phase。

**migration 006 的特殊性**

`packages/daemon/src/storage/migrations/006-conflicts-pending-review.ts`。这个 migration 要给 `conflicts` 表的 `status` 枚举增加 `PENDING_REVIEW` 值。正常思路是改 CHECK 约束，但 SQLite 不支持 `ALTER TABLE ... MODIFY COLUMN` 也不支持修改 CHECK——只能走 table-rebuild dance：

```
CREATE TABLE conflicts_new ( ... status CHECK (... 'PENDING_REVIEW') ... );
INSERT INTO conflicts_new SELECT * FROM conflicts;
DROP TABLE conflicts;
ALTER TABLE conflicts_new RENAME TO conflicts;
CREATE INDEX idx_conflicts_detected_at ON conflicts(detected_at);
CREATE INDEX idx_conflicts_status      ON conflicts(status);
```

migration 序号选 006（而非 005）是因为 005 已被上周的 `dispatch-requests` 表 migration 占用。`status` 索引是新加的——原来 004 只有 `detected_at` 索引，现在宠物 queryState() 要按 `status='OPEN'` 聚合计数，加索引让这个查询从全表扫变成 O(log n)。

**pre-commit hook 升级**

`packages/daemon/scripts/cairn-precommit-check.mjs` 从 read-only 升级为可写。原来只检测冲突行是否存在、exit 1 阻塞 commit；现在若检测到 `OPEN` 冲突，会把对应行的 status 更新为 `PENDING_REVIEW`，使宠物能区分"正常 OPEN 冲突"和"刚在 commit 前被捕获到的高优先级冲突"。

fail-open 逻辑不变：DB busy 超 2s 时静默 `exit 0`，让 commit 照常通过，不因 DB 锁阻塞正常工作流。`CAIRN_HOOK_DEBUG=1` 时才输出 stderr 日志，方便排查。

**R6 fallback rule**

在 `packages/mcp-server/src/tools/dispatch.ts` 的 `applyFallbackRules` 函数里新增第 4 条规则：rewind-recency 检测。

流程：
1. `cairn.rewind.to` 工具在执行时，调用完 git stash apply 后，向 scratchpad 写入 `_rewind_last_invoked/${agentId}`，值为当前 ISO 时间戳。
2. 下一次同 agentId 发起的 `cairn.dispatch.request`，在进入 fallback 规则链之前，读取该 scratchpad key。
3. 若距离写入时间 < 3000ms，在生成的 `generated_prompt` 末尾追加 `[FALLBACK R6] Agent recently rewound — review current file state before proceeding`。
4. 若 > 3s，或 scratchpad key 不存在，或 agentId 不同，均跳过 R6。

scratchpad key 里嵌 `agentId` 是 code review 后才修的（初版用全局键，导致跨 agent 串扰）。

sonnet 跑完汇报：daemon 216 / mcp-server 158 pass。

### 阶段四：Phase 2 — cairn.conflict.resolve + Inspector Resolve + Electron 写句柄（commit f71fbf2）

一只 sonnet 处理跨包三层改动：daemon repository 层、mcp-server MCP 工具层、desktop-shell Electron 层。

**daemon 层**

`packages/daemon/src/storage/repositories/conflicts.ts` 新增 `resolveConflict(db, id, resolution)` 方法。SQL：

```sql
UPDATE conflicts
SET status='RESOLVED', resolved_at=?, resolution=?
WHERE id=? AND status NOT IN ('RESOLVED','IGNORED')
```

WHERE 子句是 TOCTOU 守卫：若已经 RESOLVED 或 IGNORED，SQL 影响 0 行，方法返回 null。调用方据此知道这条路径是幂等的，无需重复报错。

**mcp-server 层**

新增 `cairn.conflict.resolve` MCP 工具，参数：`conflict_id: string`（required），`resolution: string`（optional，用户填写解决说明）。工具内部调用 `resolveConflict()`，返回 null 时向 LLM 回复幂等提示而非报错。

**决策 40**：Electron main process 的 DB 句柄策略拆成三层：

第一层 WAL bootstrap——`app.whenReady` 里短暂开一个 RW 句柄，执行 `PRAGMA journal_mode=WAL`，立刻关闭。目的是确保 WAL 模式在任何 read-only 句柄打开之前就已经设置好，避免跨进程（MCP server + Electron）在 journal 模式上产生冲突。

第二层 read-only 句柄——WAL bootstrap 关闭后，重新开 read-only 句柄给 `queryState()` 轮询用（每 3s 查一次 DB，驱动宠物动画）。read-only 句柄不能写，但在 WAL 模式下可以和写者并发，不阻塞。

第三层 lazy write 句柄——`resolve-conflict` IPC handler 里，只在真正被调用时才开 RW 句柄，执行完立刻关闭。Electron 主进程绝大多数时间不需要写 DB，懒初始化避免长期持有写锁。

**决策 40b**：Inspector Resolve UI 不走 MCP round-trip，原因：Inspector 是 Electron 窗口内的本地 UI，调 MCP 需要通过 IPC → main process → JSON-RPC → mcp-server 进程，链路长且调试难。直接在 `main.cjs` 的 `resolve-conflict` IPC handler 里跑与 `conflicts.ts` `resolveConflict()` 等价的 SQL，逻辑一致，不引入额外 IPC 层。UI 改动只加一个 Resolve 按钮，复用 `inspector.html/js` 现有骨架，不起新 window，不改 Electron 进程架构。

## 6. 已知未修问题（不阻塞 v0.1）

**MEDIUM**

- **install CLI 部分失败无回滚**：写完 `.mcp.json` 后若写 hook 失败，`.mcp.json` 不会回滚。手动删除成本低，但工具本应原子。v0.2 加 try/finally 清理逻辑。

**LOW style**

- **install.ts 注释偏多**：banner 注释、内联说明比 CLAUDE.md "default to no comments" 约定重，风格不统一。无功能影响，后续瘦身。

**LOW（ADR 设计意图）**

- **pre-commit hook stderr 静默 fallback**：DB busy 超 2s 时静默 `exit 0`，仅 `CAIRN_HOOK_DEBUG=1` 时打印。已是 ADR-1 fail-open 设计意图——hook 不能阻塞正常 commit，可接受。

**LOW security**

- **`ws.cwd` 含 `"` 字符路径未转义**：install 写 `.bat` / `.sh` 时路径内嵌双引号未转义。Windows 路径不含引号，Linux 合法但极罕见。v0.2 加 `shellescape` 处理。

**LOW format**

- **`ARCHITECTURE.md` §5.2 表格列对齐小不一致**：纯视觉问题，无功能影响，扣了自评 0.1 分。

**v0.2 计划项**

- **桌面像素层无自动化测试**：Electron 动画切换目前只能手动启动肉眼验证。v0.2 引入 playwright-electron 或等价方案，把"sprite 切换"纳入 CI 流程。
